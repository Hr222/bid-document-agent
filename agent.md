# agent.md

本文件是本仓库的开发规范，供后续 AI coding agent / Codex / 协作者统一遵守。

## 1. 项目定位

本项目不是单一的 RAG 工程，也不是单一的知识库工程。

本项目是一个 **Agent 工程项目**，目标是把作者过往真实业务工作中的经验、流程、规则、案例与协作方式，逐步沉淀为可复用的 agent 能力。

典型目标场景包括但不限于：

- 通过知识库 / RAG 获取财务计算规则
- 结合业务规则去对应业务数据库查取数据
- 自动生成审核表单、校验结果或辅助结论
- 结合已有内容、历史案例和既定模板生成投标书
- 将规则、数据、知识、案例和生成能力编排为可执行工作流

因此，本项目中的知识库、RAG、数据库查询、规则处理、文档生成，都是 agent 工程的一部分，而不是项目的全部。

## 2. 总体开发原则

### 2.1 先保证正确性，再考虑复杂度

- 先保证流程行为正确
- 先保证结果可验证、可解释
- 不为了“看起来高级”而过早引入复杂框架

### 2.2 小步演进

- 优先做最小闭环
- 优先在现有结构上扩展
- 尽量避免无必要的大范围重写

### 2.3 改动必须可验证

- 后端逻辑改动优先补自动化测试
- 前端改动至少保证构建通过
- 关键链路改动应保留调试可见性

## 3. 业务抽象原则

- 开发时优先抽象“业务能力”，而不是只抽象“技术组件”
- 优先思考一个能力在真实流程中的输入、规则、依赖数据、输出和验收方式
- 不要把业务问题过早简化成单一模型调用问题
- 不要把项目定位收缩成“只是一个检索问答系统”

对本项目来说，更重要的是：

- 一个 agent 如何理解规则
- 一个 agent 如何获取外部数据
- 一个 agent 如何把规则与数据结合起来形成业务结果
- 一个 agent 如何调用知识、案例和模板完成复杂文档产出

## 4. 代码组织规范

### 4.1 后端

- `app/api/routes`：接口层、参数接收、错误映射
- `app/services`：业务流程、能力编排、模型调用、规则处理
- `app/repositories`：数据库访问与查询逻辑
- `app/schemas`：请求/响应与结构化对象
- 不要把 SQL、HTTP、模型调用、复杂业务规则混写在一个模块中

### 4.2 前端

- `api/` 负责 HTTP 调用
- `types/` 负责前后端结构类型
- `components/` 负责界面组件
- 页面层负责交互串联，不要在组件里混入过多接口拼装逻辑

## 5. Agent 能力开发规范

- 每新增一个 agent 能力时，先明确：
  - 输入是什么
  - 依赖什么规则
  - 依赖什么数据源
  - 需要什么知识或案例
  - 输出是什么
  - 如何验证输出

- 优先把能力拆成清晰步骤，例如：
  - 规则获取
  - 数据获取
  - 数据清洗
  - 规则计算
  - 结果生成
  - 结果校验

- 如果一个能力包含多段处理链路，优先先做可验证的单段闭环，再做编排整合

## 6. 数据源与规则处理规范

- 规则、知识、案例、数据库，不应混为一谈
- 规则来源要明确区分：
  - 明文业务规则
  - 知识库召回结果
  - 历史案例
  - 人工配置

- 数据来源要明确区分：
  - 知识库文本
  - 业务数据库
  - 上传文档
  - 人工输入

- 在设计能力时，要明确“规则是从哪里来的，数据是从哪里来的，结果是如何计算或生成的”

## 7. RAG / 知识库使用规范

- 知识库与 RAG 是项目中的重要子能力，但不是项目全部
- 在适合的场景下使用知识库，不要强行把所有问题都改写成 RAG 问答
- 检索结果必须保留足够的来源信息，便于解释和审计
- 回答或生成结果如依赖知识库，应尽量保留引用依据

当前与知识库相关的开发应继续遵守：

- `search` 与 `ask` 使用统一 retrieval chain
- query embedding 与 document embedding 保持同源配置，除非明确变更
- 回答链路必须尽量基于检索证据

## 8. API 与工具集成规范

- 新增能力优先保持现有 API 兼容
- 修改请求/响应字段时，优先追加，不轻易破坏已有语义
- 如果接入新的数据库、工具或外部服务，要明确：
  - 它解决什么问题
  - 输入输出是什么
  - 失败时如何处理
  - 是否可替换

- 不要因为框架流行就直接引入
- 引入 LangChain / LangGraph / 新工具前，应先说明要解决的具体复杂度

## 9. 测试规范

以下改动原则上应补测试：

- repository 查询逻辑
- 规则处理逻辑
- 检索链路
- 生成链路
- 关键业务流程
- 外部依赖失败场景

前端改动至少应完成：

- `npm.cmd run build`

后端改动至少应完成：

- 相关 pytest

## 10. 日志与调试规范

- 保留必要的请求失败日志
- 对模型调用失败要明确区分本地错误和上游错误
- 对规则处理、检索、数据查询、生成等关键链路，尽量保留足够调试信息
- 调试信息至少要帮助回答：
  - 输入是什么
  - 命中了什么
  - 查了什么数据
  - 用了什么规则
  - 为什么得到这个结果

## 11. 文档规范

- 阶段总结、方案、设计说明写入 `docs/`
- `agent.md` 只保留开发规范，不记录阶段计划
- 重要能力、规则来源、数据源关系、外部依赖接入，应补文档
- 如果项目定位或核心能力边界发生变化，应同步更新文档

## 12. 配置规范

- 新增配置项时同步更新：
  - `app/core/config.py`
  - `.env.example`

- 配置命名应清晰，避免一个配置承担多种职责
- 不要把临时调试参数直接写死在代码里

## 13. 禁止事项

以下行为默认禁止，除非用户明确要求：

- 无必要重写现有主链路
- 未验证就切换关键模型或供应商
- 为了接框架而接框架
- 吞掉错误、隐藏错误来源
- 把业务问题粗暴简化成一次模型调用
- 把项目错误定义成“只是知识库项目”或“只是 RAG 项目”

## 14. 一句话要求

后续开发请始终优先保证：

**面向真实业务能力抽象，保证链路清晰、结果可验、规则可追、结构可演化。**
