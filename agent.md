# agent.md

本文档是本仓库的人机协作开发约定，供 AI coding agent、Codex 与人工协作者统一遵守。

## 1. 总体原则

- 先保证正确，再考虑复杂。
- 先做最小闭环，再逐步演进。
- 改动必须可验证，避免只改代码不做检查。
- 优先保持现有主链路稳定，避免无必要的大改。

## 2. 注释与文档要求

- **所有新加入的代码、配置、脚本、测试、文档内容，都必须补充必要的中文注释或中文说明。**
- **禁止只写英文注释。** 如果引用外部协议、字段名、专业术语，可以保留英文原词，但解释必须使用中文。
- 新增函数、类、复杂分支、关键流程、临时兼容逻辑，默认只需要说明“它是做什么的”。
- 如果逻辑特别绕、存在临时兼容、容易误解，才额外补充“为什么这样做”。
- 如果代码非常直白，可以少写注释；但只要阅读成本稍高，就必须补中文注释。
- 前端文案、调试文案、提示文案，默认优先使用中文。
- 新增或修改重要能力时，如仅靠代码不易理解，需同步补充 `docs/` 下的中文说明文档。

## 3. 后端开发约定

- `app/api/routes` 负责接口层、参数接收、错误映射。
- `app/services` 负责业务流程、能力编排、规则处理。
- `app/repositories` 负责数据库访问与查询逻辑。
- `app/schemas` 负责请求、响应与结构化对象定义。
- 不要把 SQL、HTTP、模型调用、复杂业务规则混写在同一个模块里。
- 涉及关键链路时，应保留足够的调试信息，方便定位“输入是什么、命中了什么、为什么得到这个结果”。

## 4. 前端开发约定

- `api/` 负责 HTTP 调用。
- `types/` 负责前后端结构定义。
- `components/` 负责界面展示。
- 页面层负责交互串联，不要在组件里堆过多请求拼装逻辑。
- 新增调试展示、状态展示、错误展示时，文案与说明优先用中文。

## 5. 检索与 Agent 能力约定

- 本项目不只是一个 RAG 项目，而是一个面向真实业务能力沉淀的 Agent 项目。
- 检索、规则、数据、案例、文档生成，都是 Agent 能力的一部分。
- 涉及知识库/RAG 的改动，要优先保证：
  - 链路清晰
  - 结果可解释
  - 来源可追踪
  - 结构可扩展
- `search` 与 `ask` 等相关链路，若共享同一检索过程，应尽量复用同一套 retrieval chain / pipeline。

## 6. 测试与验收约定

- 后端改动至少补相关 `pytest`。
- 前端改动至少保证 `npm run build` 通过。
- 关键业务链路改动，除自动测试外，最好补一轮手工验收说明。
- 如果因本地环境问题无法完成验证，必须明确写出“未验证项”和原因，不能默认当作已通过。

## 7. 配置与可维护性约定

- 新增配置时，需同步更新：
  - `app/core/config.py`
  - `.env.example`
- 不要把临时调试参数直接写死在业务代码里。
- 配置命名要清晰，避免一个配置承担多种含义。

## 8. 禁止事项

- 禁止新增无中文说明的复杂代码。
- 禁止只加英文注释、不加中文解释。
- 禁止为了“看起来高级”而过早引入复杂框架或复杂抽象。
- 禁止吞掉异常、隐藏错误来源。
- 禁止未经验证就声称“已经完成”或“已经通过验收”。

## 9. 一句话要求

后续协作默认遵守：

**新增内容必须有中文说明，关键代码必须有中文注释，所有改动必须可验证、可解释、可继续演进。**
