# 第三阶段 F1 工作进度

本文档是第三阶段 F1 先锋接入阶段的执行与进度记录，负责记录 F1 的实际工作状态、实现证据、验证结果和遗留问题。

第三阶段的范围、阶段关系和总体完成标准以 [第三阶段工作计划.md](第三阶段工作计划.md) 为准；本文件不替代总计划，也不扩大 F1 的工作范围。

## 1. F1 阶段定位

F1 的目标是按“技术底座 -> 独立 Agent 入口 -> 文件输入 -> 业务 Prompt”的顺序打通第三阶段的最小技术闭环：先完成 LLM 与 LangChain 接入，再建立招标书专属 Agent 入口，通过 HTTP 接收文件和查询状态，最后基于真实输入定义招标书 Prompt 与结构化输出，并确保实现符合 `agent.md` 和 `ARCHITECTURE.md` 的分层约束。

| 项目 | F1 范围 |
|---|---|
| 独立 Agent 端 | 招标书 Agent 的输入输出契约、调用入口和结构化结果返回 |
| 招标书输入 | 通过 HTTP 完成文件上传、文件准入、任务 ID、状态查询和原始来源保存 |
| LLM 接入 | 优先完成稳定的结构化 LLM Port、配置读取、失败契约、离线 Fake |
| LangChain 对接 | 优先完成基础 Adapter、模型绑定和 Structured Output 技术验证；业务 Prompt 在文件输入之后定义 |
| 投标书骨架 | 根据招标书格式生成只包含结构的投标书文档，不填写正文内容 |
| 前端 | 最小上传入口、任务状态、骨架文档结果和错误信息展示 |
| 明确不做 | 完整招标书要求理解、内容填写、知识库引用、LangGraph、SubAgent、正式文件提交 |

## 2. Agent 应用模块族定位

招标书能力属于 Agent 应用模块族下的一个独立能力，不归入现有 `modules/online`、`modules/knowledge` 或 `modules/ingestion`。后续财务、风控等能力与招标书能力并列，互不因为同属 Agent 而直接耦合。

| 位置 | F1 职责 | 不负责的内容 |
|---|---|---|
| `app/modules/agent/tender/application` | 招标书任务用例、文件接收流程、Prompt 业务流程、LLM Port 调用和结果处理 | FastAPI、LangChain SDK、文件系统和模型客户端细节 |
| `app/modules/agent/tender/domain` | 招标书任务状态、输入输出契约、结构化业务结果和状态规则 | HTTP、数据库、LLM 框架和具体适配器 |
| `app/modules/agent/tender/ports` | 招标书任务存储端口、LLM 结构化调用端口等抽象 | 具体文件存储、OpenAI SDK、LangChain 对象 |
| `app/interfaces/agent/tender` | 招标书 Agent 的输入输出契约、Agent Adapter 和调用转换 | 直接编排业务流程、直接创建 LLM 客户端 |
| `app/interfaces/http` | 文件上传、任务状态 API 的 Schema、Assembler、依赖注入和异常映射 | 直接编排 Application 之外的模型或文件流程 |
| `app/infrastructure/llm` | LLM SDK 和 LangChain Adapter 的具体实现 | 招标书业务规则和 Prompt 业务意图 |
| `app/infrastructure/filesystem` | 招标书任务文件暂存的具体实现 | 决定招标书任务状态和业务流程 |
| `app/composition/agent.py` | 组装 Agent 能力及其具体适配器 | 在模块内部隐式创建跨层依赖 |

F1 计划中的目标边界如下：

```text
interfaces/agent/tender ─┐
                         ├─> modules/agent/tender/application
interfaces/http/tender ──┘       -> modules/agent/tender/domain / ports
                                   <- infrastructure/llm / filesystem

composition/agent.py 负责统一组装
```

Agent 能力族的目标结构如下：

```text
app/modules/agent/
├── tender/                         # F1：招标书 Agent 能力
│   ├── application/
│   ├── domain/
│   └── ports/
├── finance/                        # 后续：财务 Agent 能力
└── risk/                           # 后续：风控 Agent 能力
```

## 4. Step 1：LLM 技术底座与调用契约

Step 1 只建立后续 Agent 和招标书 Prompt 可以依赖的 LLM 能力，不实现招标书业务 Prompt，不接收文件，也不生成投标书骨架。当前 Step 1 已完成代码实现和离线验证。

| 工作域 | 工作内容 | 计划落位 | 产出 |
|---|---|---|---|
| LLM 能力边界 | 明确一次结构化模型调用所需的输入、输出、模型标识、Prompt 版本和错误结果 | `modules/agent/tender/ports` | 稳定的 LLM Port 与请求/结果契约 |
| 调用配置 | 统一 API Key、Base URL、模型名、超时、温度和最大输出等配置来源 | `shared/config.py` + Composition Root | 可校验、可替换、不可泄露密钥的配置契约 |
| Provider 适配 | 通过公共 OpenAI-compatible Client Factory 复用现有 GLM 调用能力，定义真实模型与 Application 之间的转换边界 | `infrastructure/llm` | RAG 和 Agent 共享同一个 Client Factory，不向上暴露 SDK 对象 |
| LangChain 适配准备 | 确定 Chat Model、Structured Output 和调用异常如何映射到稳定 Port | `infrastructure/llm` | LangChain Adapter 的输入输出契约 |
| 结构化结果 | 定义最小技术验证结果，不代表招标书理解结果或企业事实 | `modules/agent/tender/domain` 或模块契约 | 可校验的结构化 Schema |
| 测试替身 | 提供固定成功、格式错误、超时、配置缺失和模型不可用等 Fake 场景 | `tests` | 不依赖真实外部模型的离线测试 |
| 依赖组装 | 让公共 Client Factory、生产 LLM、LangChain Adapter 和 Fake 均由 Composition Root 注入 | `composition/agent.py`、`composition/root.py` | RAG 和 Agent 可共享，生产/测试实现可切换 |

### Step 1 的调用边界

```text
Agent Application
  -> Structured LLM Port
  <- Structured LLM Result

Composition Root
  -> OpenAI-compatible Client Factory
  -> OpenAI-compatible / LangChain Adapter
  -> Fake LLM
```

| 调用方 | 可以做什么 | 不可以做什么 |
|---|---|---|
| Agent Application | 传入已经定义好的结构化请求，处理成功、失败和待处理结果 | 直接创建 `ChatOpenAI`、OpenAI Client 或读取环境变量 |
| LLM Port | 表达模型调用能力和稳定错误契约 | 依赖 LangChain 或具体供应商 SDK |
| Client Factory | 创建并缓存共享的 OpenAI-compatible Client，供 RAG 和 Agent Adapter 使用 | 决定招标书业务规则或直接编排 Application 流程 |
| LangChain Adapter | 创建 Chat Model、绑定 Structured Output、转换异常和结果 | 决定招标书业务规则或直接写入任务状态 |
| Fake LLM | 返回固定结构化结果或固定失败结果 | 访问真实网络、真实密钥或生产数据库 |

### Step 1 的技术验证用例

| 用例 | 输入 | 预期结果 |
|---|---|---|
| 正常结构化调用 | 固定测试 Prompt 和技术验证 Schema | 返回符合 Schema 的结构化结果 |
| 模型输出格式错误 | 无法通过 Schema 的模型结果 | 返回明确的结构化错误，不进入业务流程 |
| 模型调用失败 | Provider 异常或网络失败替身 | 映射为稳定的 LLM 调用失败结果 |
| 配置缺失 | 缺少 API Key、Base URL 或模型名 | 启动或调用前得到明确配置错误，不泄露敏感信息 |
| 超时 | 超时 Fake | 返回可识别的超时结果，保留重试边界 |
| 模型替换 | Fake 替换真实 Adapter | Application 测试无需修改业务代码即可运行 |

### Step 1 的明确边界

| 纳入 Step 1 | 不纳入 Step 1 |
|---|---|
| LLM Port、配置、真实 Adapter、LangChain Adapter、Fake 和契约测试 | 招标书文件接收 |
| Prompt 版本字段和结构化输出机制 | 招标书业务 Prompt 设计 |
| Agent Application 对 LLM Port 的依赖方式 | 投标书骨架生成 |
| 供应商错误、超时和格式错误映射 | LangGraph、SubAgent 和复杂工作流 |

Step 1 完成条件是：Application 可以只依赖稳定 Port，生产实现可以由 Composition Root 注入，Fake 可以覆盖主要成功与失败分支，且 Domain 不依赖 LangChain 或具体 LLM SDK。

## 6. 执行顺序与依赖

| 阶段 | 先完成的内容 | 后续内容 | 依赖关系 |
|---|---|---|---|
| 技术底座 | LLM Port、配置、Fake、LangChain Adapter | Agent 端技术契约和业务流程 | 没有稳定模型调用端口，不进入 Agent 业务 Prompt 设计 |
| Agent 入口 | 招标书 Agent 输入输出契约、Adapter 和 Composition Root 组装 | 文件输入和 Prompt 调用 | Agent 端只能调用 Application，不能直接访问 SDK、Repository 或文件系统 |
| 文件输入 | 招标书任务契约、文件暂存、来源和状态 | 招标书 Prompt 契约 | Prompt 必须基于真实文件输入和可追踪来源定义 |
| Prompt 契约 | 招标书 Prompt、版本、最小结构化输出 Schema | 投标书骨架生成 | Prompt 输出必须先通过 Schema 校验，不能直接作为业务事实 |
| 骨架生成 | 根据结构化章节和格式约束生成结构-only 文档 | 内容填写和知识库引用 | 只能生成标题、顺序、目录、表格/附件位置和占位符，不生成正文事实 |
| 交互闭环 | HTTP 上传/状态接口、骨架文档结果、前端状态展示和 Agent 结果返回 | F1 验收 | 不建设完整工作台，不把 HTTP 当作 Agent 编排入口 |

## 7. 实现边界

| 层级 | 允许负责的内容 | 禁止承担的内容 |
|---|---|---|
| Agent Interface | Agent 输入输出契约、调用转换、结构化结果返回 | 直接调用 LLM SDK、LangChain 对象、Repository 或文件系统 |
| HTTP / Frontend | 文件上传、状态查询、Schema、参数转换、依赖注入、状态展示和异常映射 | 直接调用 LLM、LangChain、Repository 或数据库 |
| Application | 招标书任务流程、Prompt 业务意图、调用端口和结果处理 | 依赖具体 SDK 或管理基础设施细节 |
| Domain / Ports | 任务状态、输入输出契约、LLM 能力端口 | 依赖 FastAPI、数据库、LangChain 或 LangGraph |
| Infrastructure | 文件存储、LLM SDK、LangChain Adapter 和具体配置 | 绕过 Application 直接改变业务状态 |
| Composition Root | 组装生产实现和测试替身 | 在各模块内隐式创建跨层依赖 |

招标书上传必须使用独立任务存储，不得调用制度知识库的正式入库流程，避免把招标书误写入现有知识库。
