# 第三阶段工作计划

本文档是 Agent Phase 3 的详细执行标准。后续 Phase 3 的开发、评审和验收以本文档为具体工作依据，同时必须遵守：

1. `agent.md`：协作与工程开发约定
2. `ARCHITECTURE.md`：架构基准和依赖边界
3. `docs/当前阶段与下一阶段计划.md`：阶段总计划
4. 本文档：Phase 3 的详细拆解、交付物和验收门槛

如果代码、文档或局部需求与上述基准不一致，必须先识别差异并完成确认，不得直接扩大当前工作范围。

## 1. 阶段定位

Agent Phase 3 的目标是：

> **理解一份具体招标书，生成符合该招标书要求的投标书应答结构，使用知识库填充有依据的内容，并由业务人员最终确认。**

核心链路：

```text
招标书输入
  -> 招标书解析与理解
  -> 当前招标书专属的投标书结构
  -> 基于证据的内容候选
  -> 人工确认、修改和标记待补
  -> 输出待确认的投标文件
```

第三阶段不是继续扩展 Phase 2 的底层检索能力，也不是直接建设适用于所有银行的通用投标书模板系统。

## 2. 阶段目标与非目标

### 2.1 阶段目标

- 接收一份具体招标书并保留原始文件和来源
- 从招标书中提取章节、要求、评分项、证明材料和格式约束
- 为当前招标书生成专属的投标书应答结构
- 使用 Phase 1 / Phase 2 的知识能力提供有依据的内容候选
- 对每条事实性内容保留来源、版本、章节、页码和原文引用
- 明确区分系统建议、原始资料、待补内容和人工修改
- 由人工确认适用性，不自动替业务人员决定最终投标内容

### 2.2 当前不做

- 通用银行投标书模板
- 跨银行格式统一
- 没有具体招标书依据的自动写作
- 未经人工确认的正式投标文件提交
- 自动判断企业资料是否适用于当前招标项目
- 一次性建设完整多场景 Agent
- 在没有真实编排需求时引入复杂 LangGraph 流程
- 直接接入正式外部业务系统

## 3. 架构和依赖约束

Phase 3 继续采用当前架构：

```text
HTTP / Frontend / Agent Adapter
  -> Application Service
  -> Domain / Ports
  <- Infrastructure Adapter

Composition Root 负责选择和组装具体实现
```

必须遵守：

- HTTP 层只负责 Schema、Assembler、依赖注入和异常映射
- 业务流程编排放在 Application Service
- 招标书要求、应答结构和审核状态等业务契约放在 Domain / Ports
- LLM、Embedding、文件、OCR、数据库和外部系统实现放在 Infrastructure
- 具体适配器由 Composition Root 组装
- Domain 不依赖 FastAPI、数据库、LLM 框架或具体外部适配器
- 前端只通过稳定 HTTP API 使用后端，不直接访问数据库、Repository 或领域对象
- 模型输出必须经过结构化 Schema 校验后才能进入业务流程
- 事实性内容必须保留来源追踪；证据不足时必须返回待补或资料不足

## 4. 分段工作计划

第三阶段按 F1-F5 推进。每个阶段完成自动化测试和人工验证后，才能进入下一阶段。

### F1：先锋接入阶段——招标书输入、LLM 与 LangChain 最小闭环

#### 目标

F1 不追求完成招标书理解，而是先验证第三阶段的最小技术闭环：招标书能够被安全接收并形成任务，LLM 能够通过稳定端口被调用，LangChain 能够在基础设施层完成 Prompt 与结构化输出适配，且所有结果都不越过现有架构边界。

| 范围 | F1 交付内容 | F1 不交付内容 |
|---|---|---|
| 招标书输入 | 上传、文件校验、任务 ID、状态查询、来源信息和原始文件保存 | 完整招标书解析、章节树、投标书工作台 |
| LLM 接入 | 结构化 LLM 调用端口、配置读取、固定模型替身和失败契约 | 依赖真实外部模型才能通过的业务测试 |
| LangChain 对接 | Prompt 模板、结构化输出、LangChain Adapter 和 Composition Root 组装 | LangGraph 工作流、SubAgent 协作、复杂 Agent 编排 |
| 投标书骨架 | 根据招标书格式生成只包含章节和格式的结构-only 投标书文档，不填写正文内容 | 完整招标书要求驱动的结构增强、企业事实填写和知识库引用 |
| 业务边界 | 上传后明确返回“等待后续招标书理解”，不伪造解析完成 | 直接生成正式投标书或自动确认企业事实 |

#### 后端工作

| 模块 | 工作内容 | 约束 |
|---|---|---|
| Application | 定义招标书任务 Command / Result、创建任务和查询状态用例 | 不在路由中编排文件、模型或 Prompt |
| Domain / Ports | 定义任务状态、任务存储端口、LLM 结构化调用端口和结果契约 | 不依赖 FastAPI、LangChain、LangGraph 或具体 SDK |
| Infrastructure | 复用文件准入与暂存能力，新增独立招标书任务存储；新增 LangChain LLM Adapter | 招标书文件不得调用制度知识库 `ingest`，避免误写知识库 |
| LLM | 接入现有配置和 OpenAI-compatible 模型端口，支持结构化 Schema 校验、超时和失败映射 | 测试默认使用 Fake，不依赖真实 API Key |
| LangChain | 使用 Prompt Template + Structured Output 完成一次最小“招标书输入能力检查/任务摘要”调用 | 仅作为 Infrastructure Adapter，不进入 Domain 和 HTTP |
| 投标书骨架 | 根据已校验的章节、顺序和格式结果生成结构-only 文档，并保留占位符 | 不生成企业事实、正文内容或知识库引用 |
| Composition Root | 统一组装任务 Repository、LLM Port 和 LangChain Adapter | 测试可注入 Fake 实现 |
| HTTP | 暴露创建任务和查询任务状态 API | 不直接调用 LangChain 或 LLM SDK |

#### 前端工作

| 前端能力 | F1 内容 |
|---|---|
| 输入 | 文件选择、格式/大小错误提示、上传操作 |
| 结果 | 展示 task_id、原始文件名、创建时间、当前状态和失败信息 |
| 骨架结果 | 在任务完成后提供结构-only 投标书文档结果，不展示为已完成的正式投标书 |
| 状态 | 至少展示 pending、processing、completed、failed、waiting_confirmation 的稳定文案 |
| 边界 | 不建设招标书章节树、内容编辑器或 Agent 控制台 |

#### F1 验收

| 验收项 | 通过标准 |
|---|---|
| 招标书输入 | 能提交合成或脱敏样例，生成唯一 task_id，原始文件可追踪且未进入制度知识库 |
| 任务状态 | 能查询稳定状态；失败时包含可理解的失败原因和重试边界 |
| 投标书骨架 | 能生成符合招标书章节顺序和基础格式的结构-only 文档；不包含企业事实、知识库内容或模型自由正文 |
| LLM 端口 | Application 只依赖稳定端口，Fake 可完整替换真实模型 |
| LangChain | Prompt 版本、结构化输出 Schema 和 Adapter 有契约测试，模型输出格式错误会被拒绝或映射为失败 |
| 架构边界 | Domain 不依赖 LangChain；HTTP 不直接调用 LLM；Composition Root 负责组装 |
| 测试 | Application、HTTP、任务存储、LangChain Adapter 和失败分支测试通过；不依赖外部模型 |
| 前端 | `npm run build` 通过，并完成上传、状态和错误展示的基本人工验证 |

### F2：招标书理解

#### 目标

从具体招标书中提取结构化要求，并保留每条要求的原文证据。

#### 首版提取范围

- 章节结构和章节顺序
- 必填章节
- 资格要求
- 技术和商务要求
- 评分要求
- 证明材料要求
- 附件要求
- 格式、页数和提交约束
- 无法确定的内容和置信提示

#### 后端工作

- 定义招标书理解输入输出 Schema
- 设计第一版 Prompt 和版本标识
- 对模型输出执行 Pydantic 或等价结构化校验
- 为每条要求保留招标书章节、页码和原文片段
- 将低置信度、冲突和无法判断的内容标记为待确认
- 通过 Application Service 编排解析、模型调用和结果校验
- 将 LLM 客户端作为 Infrastructure 能力接入

#### 前端工作

- 展示章节和结构化要求
- 展示要求对应的原文引用
- 展示低置信度、冲突和待确认项
- 不把模型自由文本直接展示为已确认业务事实

#### F2 验收

- 固定一份脱敏招标书理解样例和预期结构化结果
- 覆盖正常提取、字段缺失、模型输出格式错误和证据不足
- 每条重要要求都能回到招标书原文
- 模型不可用时有明确失败或待处理状态
- 自动化测试使用固定模型替身，不依赖不可控外部服务
- 完成一次人工复核后，才能进入 F3

### F3：招标书要求驱动的投标书结构增强

#### 目标

在 F1 结构-only 骨架的基础上，根据 F2 提取的招标书要求生成当前项目专属的投标书应答结构，而不是生成通用模板。F3 负责把资格要求、评分项、证明材料和格式约束映射到可应答节点；F1 只验证基础格式和章节骨架生成。

#### 后端工作

- 在 F1 骨架基础上，将招标书章节、要求和附件要求转换为应答结构
- 保留招标书要求的章节顺序和层级
- 创建标题、目录、应答位置和附件位置
- 标记必填、待填写、待确认和不适用候选位置
- 保留每个结构节点对应的招标书引用
- 输出结构化草稿，必要时再生成文件格式草稿

#### 前端工作

- 展示投标书章节树和章节顺序
- 展示每个章节的必填、待填和待确认状态
- 支持进入章节查看要求、引用和待填写区域

#### F3 验收

- 招标书章节顺序没有被系统擅自改变
- 必填章节、附件位置和格式约束可追踪
- 生成的是当前招标书专属结构，不是预设通用模板
- 缺少要求时明确标记，不凭常识补齐
- 结构生成有稳定的快照或契约测试

### F4：内容填写

#### 目标

使用知识库和 Phase 2 已完成的规则、数据、检索与引用能力，为应答结构提供有证据的内容候选。

#### 后端工作

- 根据章节和招标书要求构造检索任务
- 复用混合检索、rerank、RAG 引用和来源追踪
- 使用文档、版本、章节、页码、原文和文件来源生成内容候选
- 对企业事实只使用有证据的资料
- 对没有资料依据的字段输出待补，不生成假设事实
- 记录内容候选的来源、生成方式、置信信息和缺失原因
- 保留系统建议与人工修改前后的版本

#### 前端工作

- 展示内容候选和原始引用
- 展示资料版本、文件来源和章节页码
- 展示待补字段、证据不足和待确认状态
- 支持人工编辑，但不覆盖原始系统建议和引用

#### F4 验收

- 内容候选可以回溯到知识库证据
- 无证据时不会编造企业事实
- 招标书要求的结构不会被内容填写流程擅自改变
- 覆盖有证据、无证据、多个候选和资料冲突场景
- 内容候选和引用契约有应用层回归测试

### F5：人工确认

#### 目标

由业务人员确认招标书理解结果、资料适用性和最终填写内容。

#### 系统负责

- 展示待确认项
- 展示招标书原文引用和知识库资料引用
- 保留系统建议、人工修改和最终确认结果
- 支持确认、修改、退回和标记待补
- 输出最终待确认文件或文件草稿

#### 系统不负责

- 替代业务人员判断资料适用性
- 自动确认企业事实
- 自动提交正式投标文件

#### F5 验收

- 人工可以区分系统建议、原始资料和人工修改
- 每个确认项有状态和操作记录
- 修改前后内容可追踪
- 未确认项不会被标记为最终完成
- 输出文件包含待补和待确认提示

## 5. LangChain、LangGraph 与 SubAgent 引入策略

### 5.1 F1：完成 LangChain 最小技术接入，不引入复杂 Agent 编排

F1 需要完成 LLM 和 LangChain 的最小接入，但不建设完整招标书理解流程。F1 的 LangChain 只承担技术适配职责，用于验证 Prompt、模型调用和 Structured Output 能够在当前架构中稳定工作。

| F1 允许接入 | F1 明确不做 |
|---|---|
| `ChatPromptTemplate` 或等价 Prompt 模板 | LangGraph 状态图 |
| `with_structured_output` 或等价结构化输出 | 多 SubAgent 路由、协作和循环 |
| `langchain-openai` 的 OpenAI-compatible Chat Model 适配 | 长流程恢复、人工暂停后续跑和复杂重试编排 |
| 文档输入上下文的最小拼装、结构-only 投标书骨架生成 | 完整招标书要求理解、企业内容填写和知识库引用 |

Prompt 的业务意图、输入来源和 Application 流程仍由应用层定义；LangChain 的具体对象、模型调用和输出转换由 Infrastructure Adapter 实现。

### 5.2 F2：在 F1 技术接入稳定后进行招标书理解

F2 才开始把招标书文本送入结构化理解流程。F2 可以复用 F1 的 LLM Port、Prompt 版本机制、Schema 校验、失败处理和 LangChain Adapter，但不能跳过 F1 的契约测试与离线替身验证。

### 5.3 什么时候引入 LangGraph

只有出现真实的状态编排需求时，才引入 LangGraph，例如：

- 招标书理解、结构生成、内容填写和审核之间存在条件分支
- 需要保存中间状态并支持暂停后恢复
- 需要人工确认后继续执行
- 节点失败后只重试或回退某一步
- 多个 SubAgent 需要路由、协作或循环校验
- 任务执行时间较长，需要可观察、可恢复和可审计

LangGraph 负责流程图、状态和节点编排；LangChain 负责模型、Prompt、Retriever 和 Tool 等组件。两者不是互相替代关系。

### 5.4 SubAgent 使用标准

- 单个 SubAgent 被 Application Service 顺序调用：不要求 LangGraph
- 多个 SubAgent 之间有明确分工但流程仍然线性：可先使用 Application Service + LangChain
- 多个 SubAgent 需要共享状态、动态路由、循环、失败恢复或人工确认：评估 LangGraph
- SubAgent 必须通过稳定的输入输出契约工作，不能直接读写数据库或绕过 Application 层
- 每个 SubAgent 的结果必须经过结构化校验，并保留来源和执行 trace

### 5.5 框架边界

推荐的依赖方向：

```text
Application Service
  -> LLM / Agent / Workflow Port
  <- LangChain / LangGraph Adapter
  <- Infrastructure Technical Adapter
```

LangChain / LangGraph 不得成为 Domain 的直接依赖，也不得让 HTTP 路由直接编排模型、工具和工作流。

## 6. 测试与验收标准

每完成一个 F 阶段，必须同时完成：

- 对应的 Application 单元测试
- 关键 HTTP API 测试
- 正常、缺失、失败和证据不足分支测试
- 外部模型和工具的稳定替身测试
- 结果来源追踪回归测试
- `ruff check app tests`
- `python -m compileall -q app tests`
- 前端变更对应的 `npm run build`
- 必要的人工操作验证

如果引入 LangChain 或 LangGraph，还必须补充：

- Prompt 或 Graph 版本固定测试
- 节点输入输出契约测试
- 状态恢复和失败回退测试
- 人工确认节点测试
- 不依赖真实外部模型的离线回归测试

未完成自动化测试和人工验证，不得将对应 F 阶段标记为完成。

## 7. 当前执行顺序

当前只推进 F1：

| 顺序 | 工作包 | 结果 |
|---|---|---|
| 1 | 定义 LLM Port、LangChain Adapter 和 Agent 端结构化契约 | 技术底座与独立 Agent 入口契约稳定 |
| 2 | 实现 LangChain 基础模型调用和 Structured Output 验证 | 完成 LLM / LangChain 技术接入验证 |
| 3 | 选定合成或脱敏招标书样例并定义任务契约 | 固定 F1 输入样例，不接入正式业务资料 |
| 4 | 实现独立任务暂存与状态查询 | 文件来源可追踪，避免写入制度知识库 |
| 5 | 基于文件输入定义招标书 Prompt 和结构化结果 | 业务 Prompt 有真实输入依据 |
| 6 | 生成结构-only 投标书骨架文档 | 输出章节、顺序、格式和占位符，不填写正文 |
| 7 | 暴露 HTTP 辅助 API 并完成前端上传、状态和骨架结果展示 | 形成最小可操作闭环 |
| 8 | 完成离线测试、架构测试、构建和人工验证 | 满足 F1 验收门槛后再进入 F2 |

F1 完成后再进入 F2。F2 完成并经过人工复核后，才进入 F3。

## 8. 阶段完成定义

第三阶段完成的最低标准是：

- 能理解具体招标书，而不是依赖通用模板
- 能生成当前招标书专属的应答结构
- 能基于知识库证据提供内容候选
- 能明确标记资料不足、冲突和待确认项
- 能保留招标书、知识库和人工修改的来源链路
- 能由业务人员确认、修改和完成最终输出
- 全流程符合 `agent.md` 和 `ARCHITECTURE.md`

第三阶段不以“模型能生成一份看起来完整的投标书”作为完成标准，而以结构符合招标书、事实有来源、过程可追踪、结果可人工确认作为完成标准。
