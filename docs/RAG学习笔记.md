## 一、我对RAG的理解

RAG，全称是 `Retrieval-Augmented Generation`，用一个更容易理解的话来说，就是：

**知识检索增强生成。**

1. 两个关键词:

| 定义           | 说明                    |
|--------------|-----------------------|
| `Retrieval`  | 先去知识库里把和当前问题最相关的内容找出来 |
| `Generation` | 再让大模型基于这些找到的内容去组织回答   |

2. 之所以需要RAG, 是因为LLM推出是有滞后性的, 对于推出以后的发生的事情是不知道, 同时许多公司并不会将敏感数据放上网络因此才需要RAG. 下面是我之前工作上遇到的问题清单:

| 问题           | 存在风险点                     |
|--------------|---------------------------|
| 知识不是最新的      | 模型参数里的知识不是实时更新的           |
| 不了解企业私有资料    | 公司制度、收费标准、内部规范并不在模型预训练语料里 |
| 容易“说得像真的”的"幻觉" | 即使不知道，也可能生成一段看起来很像答案的话    |
| 很难回溯依据       | 只靠模型生成，很难说明答案是从哪份资料来的     |

所以思路很清晰, RAG的出现就是解决下面这些场景的:

| 场景问题        | RAG 的作用 |
|-------------|---|
| 模型不知道企业内部信息 | 先检索制度内容，再回答 |
| 需要回答时给出出处   | 让回答建立在命中的原文片段上 |
| 需要减少幻觉      | 用真实资料约束模型生成 |
| 需要后期维护知识    | 更新知识库，而不是重新训练模型 |

 RAG 其实不是“让模型更聪明”，而是“让模型更有依据”。

**让大模型的回答，尽量建立在可检索、可引用、可回溯的知识基础上。**

## 二、向量数据库

### 2.1 一个具体场景

抛出一个场景。

比如在公司规章制度里写的是：

> 员工离职后仍应承担保密义务

但用户真正问的可能是：

- 员工离职以后还要不要保密？
- 离职后的保密责任怎么算？
- 保密义务会不会因为离职而结束？

这时候就会发现一个问题：

**用户的问题和原文在字面上可能并不完全一致，但语义上其实是在问同一件事。**

若是靠传统的精确匹配，不管是 `like`、`equal`，还是普通条件查询，它们更擅长解决的都是“这几个字有没有出现”“这个字段是不是等于某个值”。

在agent真实问答场景里，很多时候用户不会按照原文来提问，而是会用更口语化、更模糊、甚至更接近自己理解的方式来表达问题。也正因为这样，仅仅把资料存起来是不够的，更关键的是系统能不能把“意思接近”的内容找出来。

向量数据库并不是简单地把一段文字原样存进去，而是会先把文本转换成能够表达语义特征的向量，再基于向量之间的距离或相似度去做检索。

换句话说：

| 方式 | 更擅长解决什么问题 |
|---|---|
| 关系型数据库 | 精确存储、结构化查询、条件过滤 |
| 关键词检索 | 查有没有出现某些词 |
| 向量数据库 | 查语义上是不是相近 |

### 2.2 为什么需要向量数据库

向量数据库的价值，不只是“存向量”这三个字，而是它让系统有机会去做一件传统精确匹配很难做好的事情：

**按语义相似度去找内容，而不只是按关键词完全匹配去找内容。**

基于上表可以知道传统数据库更适合做结构化管理、状态维护、条件筛选；向量数据库更适合做语义召回。

这两者在 RAG 场景里并不是互相替代的关系，而更像是互相配合的关系。

再往前理解一步，向量数据库是 RAG 检索层里一块非常关键的基础设施。它解决的核心问题不是“回答”，而是“先把最相关的内容找出来”。

也就是说：

- 向量数据库负责“找”
- 大模型负责“答”
- RAG 负责把“找”和“答”接起来

所以为什么很多 RAG 系统都会搭配向量数据库，本质上不是因为这是一种流行搭配，而是因为真实业务里的很多问题，本来就不是靠关键词精确匹配就能解决的。

尤其在制度、规范、业务规则这类资料里，这种场景下感觉会更明显。用户往往不会按文档原句发问，而是会带着自己的理解、口语化表达甚至模糊印象来问。这时如果没有语义检索，只靠精确匹配，检索效果通常会比较僵硬。

所以到现在为止，最核心的理解是：

**向量数据库解决的是“怎么更好地找”，RAG 解决的是“怎么基于找到的内容去回答”。**

### 2.3 从概念到落地

明白了向量数据库定义后, 在落地前还有两个更实际的问题：

1. 资料到底应该怎么切片
2. 切完以后，这些内容要以什么结构存进向量表

这两件事其实是连在一起的。

如果前面没有先想清楚“怎么切”，后面就很容易变成只是把一堆文本做成向量存进去；但这些向量到底对应的是整篇文档、某一章、某一条，还是某一段，其实是模糊的。这样做出来的检索效果，往往也不会稳定。

### 2.4 对切片通用处理(个人经验)

对切片并不能是简单地把文档按固定字数拆开，而是把它看成一个围绕业务场景去设计检索单元的过程。切片的核心目的，不是单纯为了把长文拆短，而是为了让系统在检索时既能尽量找准内容，又不要丢掉必要的上下文。

切片没有一个绝对统一的标准。切得太小，容易把原本完整的语义打碎；切得太大，又容易把多个主题混在一起，让相似度检索不够精准。切片本质上是在平衡两件事：

| 平衡点 | 过小会怎样 | 过大会怎样 |
|---|---|---|
| 语义完整性 | 上下文不完整，容易断章取义 | 相对完整 |
| 检索精度 | 精度可能高，但容易碎 | 容易混入无关内容 |

PS: 以前自己刚上手用过的一个土方法，按场景去切，先做一个大类的切片规则，然后交给程序去跑。虽然不够规范，但方向大致其实是对的。因为有一部分是人工参与但是不够智能就是了, 很不ai工程化

因为切片本来就不是纯技术动作，它一定带着业务理解。更准确一点说，应该是：

**人工负责定义切片策略，程序负责按策略批量执行。**

| 层 | 谁负责 | 做什么 |
|---|---|---|
| 策略层 | 人 | 根据业务场景定义这类资料该怎么切 |
| 执行层 | 程序 | 按规则批量解析、切块、清洗、入库 |

根据之前踩过的坑, 就个人理解真正决定“怎么切”的，还是业务场景、文档结构和问答目标。

比如：

| 文档类型 | 更适合的切片思路 |
|---|---|
| 制度类 | 优先按章、条、款切，必要时再按长度补切 |
| 收费标准类 | 按项目、子项、表格行切 |
| 合同模板类 | 按条款、责任、付款条件切 |

拿制度类资料来说，应该把“条”款作为核心切片单位。因为“条”通常已经接近一个相对完整的规则表达，既比整篇文档更适合检索，也比一句一句切更不容易把语义打碎。

所以到现在为止，我对切片最核心的理解是：

**RAG 的切片策略是由业务场景、文档结构和问答目标共同决定的；人工负责定义规则，程序负责按规则执行。**

### 2.5 如何解建立向量表

对“建立向量表”并不是“建一张表存 embedding”这么简单。更准确地说，向量表是 RAG 检索层里的核心数据结构，它保存的并不是普通业务数据，而是**经过切片之后、可以被语义检索的知识单元**。

在建立向量表之前至少要先想清楚三个问题：

| 问题 | 本质上在解决什么 |
|---|---|
| 我到底在检索什么 | 检索单位是什么 |
| 我靠什么判断两段内容语义接近 | 相似度计算靠什么完成 |
| 命中之后怎么回到原文 | 内容如何定位和溯源 |

也就是说，向量表里真正重要的信息，至少不是只有一列向量，而是至少要包含下面几类内容：

| 信息类型 | 作用 |
|---|---|
| 文本内容 | 这是未来真正被检索、也会被送给模型的上下文 |
| 向量表示 | 这是用来做语义相似度匹配的核心 |
| 来源信息 | 这是命中后回到原文、章节、页码、制度名称的依据 |
| 模型信息 | 这是后续重建向量、切换 embedding 模型的重要基础 |

`document` 更像知识来源单位，它回答的是“这份资料是什么”；而 `chunk` 更像检索单位，它回答的是“系统实际拿哪一段内容去做召回”；`embedding` 则是 `chunk` 的语义表达方式，它负责让系统能够按“意思接近”而不只是“字面相同”去找内容。

具体如下：

| 单位 | 更像什么 | 在 RAG 里主要承担什么作用 |
|---|---|---|
| `document` | 知识来源单位 | 表示这份资料本身是什么 |
| `chunk` | 检索单位 | 表示系统实际拿什么内容去召回 |
| `embedding` | 语义表达单位 | 表示这段内容如何参与相似度计算 |

所以建立向量表，本质上不是简单存一列向量，而是在设计：**一个切片后的知识单元，如何被表示、被检索、被定位、被维护。**

### 2.6 我现在怎么理解向量表字段和场景

如果只保留最小骨架，向量表至少应该围绕下面几类字段来设计：

| 字段类型 | 作用 | 为什么重要 |
|---|---|---|
| `id` | 向量记录主键 | 唯一标识一条检索记录 |
| `document_id` | 所属文档 | 知道这段内容来自哪份资料 |
| `chunk_id` | 所属切片 | 知道它对应哪个检索单元 |
| `chunk_text` | 切片文本 | 检索命中后真正给模型看的内容 |
| `embedding` | 向量值 | 用来做语义相似度计算 |
| `model_name` | 向量模型名 | 方便后续模型切换和重建 |

如果再往工程化一点走，下面这些字段是具有参考价值的：

| 字段类型 | 作用 |
|---|---|
| `section_title` | 记录它属于哪一章、哪一节 |
| `article_no` | 记录它对应第几条、第几款 |
| `category` | 区分制度、收费标准、合同等类型 |
| `source_type` | 区分 PDF、DOCX、人工录入 |
| `page_no` | 方便 PDF 场景回溯原页 |
| `version` | 支持文档更新和向量重建 |
| `status` | 控制是否参与检索 |
| `created_at` | 记录生成时间 |

这样再看，向量表里真正保存的，其实不只是向量，而是：

| 信息层 | 保存的内容 |
|---|---|
| 文本层 | 切片后的文本内容 |
| 向量层 | 文本对应的 embedding |
| 来源层 | 文档、章节、条款、页码等定位信息 |
| 维护层 | 模型、版本、状态、时间等管理信息 |

PS: 需要什么元数据字段是需要场景支撑的。

比如：

| 场景 | 为什么要有对应字段 |
|---|---|
| 用户问制度里的某一条规定 | 需要 `chunk_text`、`article_no`、`section_title` |
| 用户问收费标准某个项目 | 需要 `category` 区分制度和收费资料 |
| 检索结果出来后要展示原文来源 | 需要 `document_id`、`page_no`、`source_type` |
| 后面更换 embedding 模型 | 需要 `model_name`、`version` |
| 文档更新后要重建向量 | 需要版本和状态字段 |
| 只想检索“当前有效”的资料 | 需要 `status` 或有效性标记 |

所以字段设计本质上不是孤立的技术动作，而是由后面的检索、展示、维护场景反推出来的。

### 2.7 向量索引

同关系型数据库一样, 向量数据库一样要面对数据越来越多的这个现实的问题：

**这些向量到底怎么更快地被找出来。**

切片决定检索单位，向量表决定这些单位怎么保存，而向量索引决定系统怎样在这些向量里更快地找到相近候选。

| 环节 | 解决的问题 |
|---|---|
| 切片 | 决定检索单元是什么 |
| 向量 | 决定语义怎么表达 |
| 向量表 | 决定这些内容怎么存储和管理 |
| 向量索引 | 决定怎么更快找到相近候选 |
| RAG | 决定怎么把候选内容交给大模型生成回答 |

向量索引和传统关系型数据库里的索引的服务的目标, 适用场景:

| 对比维度 | 传统关系型索引 | 向量索引 |
|---|---|---|
| 核心目标 | 加速精确查询、范围查询、排序过滤 | 加速相似度检索、近邻搜索 |
| 面向的数据 | 结构化字段值 | 向量化后的语义表示 |
| 常见判断方式 | 是否等于、是否在范围内、排序顺序 | 距离是否更近、语义是否更相似 |
| 更擅长解决的问题 | `id = ?`、时间范围、状态过滤、唯一定位 | “这段话和用户问题意思是不是接近” |
| 检索特征 | 更强调精确命中 | 更强调相似候选召回 |

两者的定位区分：

| 索引类型 | 更像在负责什么 |
|---|---|
| 关系型索引 | 负责结构化过滤和精确定位 |
| 向量索引 | 负责语义召回和相似度搜索 |

比如在实际检索时，一个比较自然的流程往往是：

1. 先用结构化条件缩小范围
2. 再在这个范围里做向量相似度检索

这时候前者靠的是传统数据库索引，后者靠的是向量索引。也就是说，关系型索引负责“筛”，向量索引负责“找近似”。

所以到现在为止，我对向量索引最核心的理解是：

**它并不是为了改变知识怎么存，而是为了在知识已经向量化之后，让系统能够更快地完成语义相似度召回。**

PS: 索引我默认选择性能不错的HNSW, IVF并不怎么使用

## 三、实践操作

一般的操作顺序是采用一条通用流水线(个人经验之谈), 如下表：

| 阶段 | 名称 | 输入 | 输出 | 是否通用               |
|---|---|---|---|--------------------|
| 1 | 文件登记 | 原始文件 | 文件身份信息 | 是                  |
| 2 | 文本解析 | 文件身份信息 | `raw_text` | 否，通常按文件类型适配不同的操作策略 |
| 3 | 文本清洗 | `raw_text` | `clean_text` | 是                  |
| 4 | 章节拆分 | `clean_text` | `sections` | 是                  |
| 5 | 切块 | `sections` | `chunks` | 是                  |
| 6 | 向量化 | `chunks` | `embeddings` | 是                  |
| 7 | 数据入库 | 文件、章节、切块、向量 | 数据库记录 | 是                  |
| 8 | 检索问答 | 用户问题 | 检索结果 + 回答 | 是                  |
| 9 | 效果验证 | 问题集 | 召回观察结果 | 是                  |

项目落地分层处理如下:

| 层级 | 作用 | 说明 |
|---|---|---|
| 文件接入层 | 接收原始文件 | 记录路径、类型、文件名、hash |
| 解析适配层 | 提取文本 | `docx`、`pdf` 在这里分流 |
| 统一文档层 | 标准化解析结果 | 将不同解析器输出统一为同一种中间结果 |
| 文本处理层 | 清洗、拆分、切块 | 这里开始不再关心文件类型 |
| 检索层 | embedding 与召回 | 面向 chunk 工作 |
| 问答层 | LLM 回答与引用 | 面向检索结果工作 |

## 四、文本清洗原则

遇到DOCX或者PDF文件, 都会遇到一些文档的空白或无意义的内容, 这些应当进行适当的内容清晰, 但清洗应该偏保守，不应该为了“看起来更整齐”而破坏制度原文结构。

建议做的清洗：

| 操作 | 是否建议 |
|---|---|
| 去多余空白 | 是 |
| 去重复页眉页脚 | 是 |
| 去页码噪音 | 是 |
| 统一换行 | 是 |
| 统一全半角 | 是 |
| 保留章、条、款编号 | 必须 |

不建议现在就做的处理：

| 操作 | 是否建议 |
|---|---|
| 改写句子 | 否 |
| 删除看似重复内容 | 否 |
| 把列表强行压成一段 | 否 |
| 重新编排正文层级 | 否 |

## 五、 MVP 阶段验收

在验证路径时候应当先跑通通路，尽可能收窄范围：

| 项目 | 当前建议 |
|---|---|
| 首批样本数量 | 10 到 20 份 |
| 首批优先文件类型 | `docx` |
| 次优先文件类型 | 原生文本 `pdf` |
| 暂不处理 | 扫描件、`.doc`、图片附件 |
| 检索目标 | 能稳定召回制度条款 |
| 问答目标 | 能基于召回结果回答并给出处 |

## 六、效果验证

RAG 不能只停留在“能跑”，第一阶段就应该同步准备验证方式。

建议至少准备一小批问题集，用来观察：

| 验证项 | 关注点 |
|---|---|
| 是否能命中正确文档 | 文档级召回是否靠谱 |
| 是否能命中正确章节 | section 级定位是否靠谱 |
| 命中 chunk 是否相关 | chunk 切分是否合理 |
| 问答是否能回溯 | 回答是否能找到原文依据 |

## 七、验收阶段

这一阶段最重要的不是做复杂的 Agent 或工作流编排，而是先把下面这件事做扎实：

**任意样板文档 -> 统一解析结果 -> 清洗 -> 章节拆分 -> chunk -> 向量化 -> 入库 -> 检索问答 -> 召回验证**

如果这条通路稳定下来，后续再接：

- 前端知识库维护界面
- 检索调试页
- LLM 问答页
- LangGraph / LangChain 流程编排

都会顺很多。

## 八、RAG 检索优化

当 RAG 最小闭环已经跑通后，后续学习重点通常会从“能不能检索”逐步转向“检索效果能不能更稳、更准、更可控”。这部分内容仍然属于 RAG 范畴，但已经开始进入检索优化阶段。

### 8.1 什么是召回

在检索语境里，`召回` 更像是在说：

**先把“可能相关”的候选内容找出来。**

它不要求第一步就一定找到“最终最完美答案”，而是先保证相关候选不要漏掉太多。

可以简单理解为：

- 回答阶段负责“组织答案”
- 召回阶段负责“先找候选”

在 RAG 里，召回做得不好，后面的 LLM 再强也很难补回来。因为模型回答所依赖的依据，本来就没有被找出来。

### 8.2 什么是向量召回

`向量召回` 是当前 RAG 里最常见的一类召回方式。

它的思路是：

1. 把用户问题转成 embedding
2. 把知识库里的 chunk 也保存成 embedding
3. 通过相似度比较，找出语义最接近的内容

它更擅长解决的问题是：

- 用户提问和原文不完全同字面
- 用户表述更口语化
- 问题和原文“意思接近”但措辞不同

比如：

- 原文：`员工离职后仍应承担保密义务`
- 用户提问：`离职以后还需要继续保密吗`

这种场景下，向量召回通常比纯关键词匹配更自然。

### 8.3 什么是关键词召回

`关键词召回` 更接近传统检索思路，它关注的是：

**原文里有没有出现这些词。**

它更适合的场景通常包括：

- 制度编号
- 专有名词
- 明确条款词
- 特定字段值

比如用户问：

- `三级审批`
- `第八条`
- `采购管理制度`

这类内容如果只靠向量检索，有时不一定最稳定；而关键词召回在字面命中上通常会更直接。

所以可以这样理解：

| 召回方式 | 更擅长什么 |
|---|---|
| 向量召回 | 找“意思相近” |
| 关键词召回 | 找“字面命中” |

### 8.4 什么是混合检索

`混合检索` 指的是：

**把向量召回和关键词召回结合起来使用。**

它的目的不是为了让系统显得更复杂，而是为了同时兼顾：

- 语义理解能力
- 字面命中能力

一个常见思路是：

1. 先做向量召回
2. 再做关键词召回
3. 把两边结果合并
4. 再进入后续排序或生成

这类方式特别适合：

- 制度类资料
- 规则类资料
- 条款类资料
- 带编号、名称、专有词的资料

因为这些资料既有“语义理解”需求，也有“精确字面命中”需求。

### 8.5 什么是 rerank

`rerank` 可以理解成：

**先粗找一批结果，再对这批结果重新排一次序。**

它通常出现在召回之后，作用是把真正更相关的内容尽量排到前面。

可以把它看成两段式：

1. 第一段：先尽量多找一些候选
2. 第二段：再从这些候选里重新判断谁更相关

它要解决的问题往往不是“有没有找到”，而是：

- 找到的结果顺序对不对
- 最相关的内容是否排在前面
- 后续给 LLM 的证据是否足够干净

### 8.6 常见检索算法类别与适用场景

当开始接触检索优化后，很容易先记住某一个具体名词，比如 `BM25`、`bge-reranker`、`embedding` 检索，但真正更重要的是先建立一层分类意识：

**这些方法分别属于哪一类，它们各自主要解决什么问题。**

如果没有这层分类意识，后面就容易出现一种情况：

- 知道很多术语
- 但一到项目里，不知道该选哪条路

所以我现在更愿意先把常见方法压缩成几张表来看。

#### 1. 主表：类别、解决问题、适用场景

| 类别 | 代表方法 | 更主要解决什么问题 | 更适合什么场景 |
|---|---|---|---|
| 关键词检索类 | `BM25`、`TF-IDF`、布尔检索 | 字面命中问题 | 制度名、编号、专有词、条款号、明确字段值 |
| 稀疏学习检索类 | `SPLADE`、`uniCOIL` | 更强的稀疏关键词检索问题 | 希望保留关键词检索特征，但又想比传统 BM25 更灵活 |
| 稠密向量检索类 | `embedding` 检索、`DPR`、`BGE`、`E5` | 语义相似问题 | 口语化提问、意思相近表达、企业问答、FAQ |
| 混合检索类 | 关键词检索 + 向量检索 + 融合 | 多种召回方式互补问题 | 制度条款问答、规则类文档、既要语义理解又要字面命中 |
| 重排类 | `Cross-Encoder reranker`、`BGE Reranker`、`Cohere Rerank` | 候选排序准确性问题 | 已经能召回，但顺序不稳，top3 有了但 top1 不准 |

#### 2. 对比表：优点与短板

| 类别 | 主要优点 | 主要短板 |
|---|---|---|
| 关键词检索类 | 简单直接、可解释性强、对编号和专有词稳定 | 对口语化问法不够自然，换一种说法就可能不稳 |
| 稀疏学习检索类 | 比传统关键词检索更灵活，仍保留一定可解释性 | 工程复杂度更高，通常不适合最早期 MVP |
| 稠密向量检索类 | 擅长理解语义相近表达，对口语化提问更自然 | 编号、条款号、专有词命中不一定最稳，可能“意思接近但不是最想要那条” |
| 混合检索类 | 能同时兼顾语义理解和字面命中，真实业务里更稳 | 链路更复杂，需要处理融合、去重、调试和评测 |
| 重排类 | 能把更相关结果排到前面，提升 top1 质量 | 不能凭空补召回，候选里没有正确结果时无能为力 |

#### 3. 场景表：先按“遇到什么问题”来选

| 场景 | 更优先考虑哪类方法 |
|---|---|
| 制度名、编号、专有词、条款号 | 关键词检索类 |
| 口语化问答、意思相近表达 | 稠密向量检索类 |
| 制度条款问答、规则类检索 | 混合检索类 |
| 已经能召回但排序总不稳 | 重排类 |
| 希望关键词检索更强，但又不想只靠传统统计规则 | 稀疏学习检索类 |

#### 4. 我现在对这层知识的理解

到这里，一个更稳的理解方式应该是：

**先不要急着记所有算法名，而是先记住“每一类方法在解决什么问题、适合什么场景、优点和短板分别是什么”。**

因为在真实项目里，真正决定选型的，通常不是“这个算法名字听起来高级不高级”，而是：

- 当前问题到底是召回不足
- 还是字面命中不足
- 还是语义理解不足
- 还是排序不准

把这个问题先判断清楚，再去选算法，思路才会更稳。

## 九、RAG 工程化

当 RAG 不再只是一个临时脚本，而是要持续调试、持续优化、持续扩展时，就会进入更偏工程化的阶段。

这部分关注的重点不再只是“RAG 原理”，而是：

- 系统结构怎么设计
- 检索链路怎么抽象
- 怎么验证改动有没有变好
- 怎么给后续混合检索、rerank、LangChain 留接口

### 9.1 什么是 retrieval pipeline

`retrieval pipeline` 可以理解成：

**系统完成检索这件事的一条步骤化链路。**

例如一条最简单的 pipeline 可能是：

1. 用户提问
2. 问题向量化
3. 向量召回
4. 分数过滤
5. 返回结果

以后如果要扩展，也可能变成：

1. 用户提问
2. 向量召回
3. 关键词召回
4. 结果融合
5. rerank
6. 返回结果

所以 pipeline 的核心意义在于：

**把检索流程组织成可观察、可插拔、可扩展的步骤。**

### 9.2 什么是 retriever

`retriever` 可以简单理解为：

**一个专门负责“帮系统找资料”的模块。**

它不负责生成答案，它只负责根据问题去知识库里找候选内容。

在工程化里把 retriever 单独看待的好处是：

- 检索逻辑更清楚
- 后续更容易替换策略
- 更方便接入 LangChain 等框架

### 9.3 为什么要暴露中间调试信息

如果系统最后只返回“结果”，但看不到中间过程，就很难判断问题出在哪。

所以在检索调试里，通常希望能看到一些中间信息，例如：

- 用了哪种检索策略
- 一共召回了多少条
- 过滤前后各剩多少条
- 每条结果来自哪一路召回
- 每条结果的分数怎么来的

这样做的价值是：

- 方便定位问题
- 方便比较不同检索策略
- 方便后续做混合检索和 rerank 实验

### 9.4 什么是评测集

`评测集` 可以理解成：

**一组提前准备好的标准问题，用来稳定测试检索或问答效果。**

比如可以准备一批问题：

- 某个问题应该命中哪篇制度
- 应该命中哪个章节
- 理想情况下应该命中哪条 chunk

它的作用是让验证不再只靠“临时试几个问题看看”，而是有一组可以重复使用的标准样本。

### 9.5 什么是 baseline

`baseline` 是一个非常常见但很重要的工程化概念。

它的意思是：

**先记录当前版本的效果，作为以后对比的基准线。**

比如当前系统在 20 个问题上的表现是：

- 15 个能命中正确文档
- 11 个能命中正确章节
- 9 个能把最相关 chunk 放在前 3 条

这就是一份 baseline。

以后你改了检索逻辑，就拿新结果和 baseline 比。这样你才知道：

- 是真的优化了
- 还是只是代码更复杂了

### 9.6 为什么评测要前置

很多项目一开始容易先堆功能，等做完再回头看“效果怎么样”。但这样往往会有一个问题：

**改了很多东西，却不知道到底有没有变好。**

所以评测前置的意思是：

- 先准备问题集
- 先记录 baseline
- 再开始改检索

这样每做一次优化，都能更清楚地判断：

- 召回率有没有提升
- 排序有没有更合理
- 新策略有没有带来副作用

对 RAG 来说，这一步很关键。因为检索优化往往不是“看起来更高级”就算成功，而是必须能被验证。
