# Phase 3 业务扩展风险与技术债参考

本文档记录 Agent Phase 3 在扩展业务能力时需要持续关注的风险和技术债。

它不是当前阶段的实施计划，也不是架构基准或验收文档。它的作用是：当 Phase 3 从首个应用样例扩展到更多场景、数据源和规则类型时，提供回看依据，避免把 Phase 2 的 PoC 实现误认为已经完成通用化。

## 1. 当前基线

当前项目已经完成：

- Agent Phase 1：资料解析、清洗、OCR、入库和知识底座建设
- Agent Phase 2：混合检索、rerank、HNSW 准备、规则/数据/结果 PoC 和知识库收尾
- 架构边界治理：HTTP、Application、Ports、Infrastructure 和 Composition Root 的依赖方向已通过测试验证
- 首个规则场景：委托评估机构申请材料核验

当前实现已经满足 Phase 2 的验证目标，但规则场景、数据 Provider 和决策输出仍以首个 PoC 为主要样本。以下内容属于业务能力扩展风险，不阻塞 Phase 2，也不代表当前代码存在故障。

## 2. 已识别的扩展风险

### R1：规则内容与执行框架仍有耦合

Checklist 执行骨架已经抽象，但当前规则定义仍主要位于具体场景的 Python 代码中。

风险：

- 新增场景或调整规则时需要修改 Python 代码
- 规则内容无法独立于执行框架进行版本管理
- 规则变更的审核、发布和回滚能力不足

触发条件：开始维护第二个以上规则场景，或规则需要由业务人员频繁调整时，应优先考虑规则外置化。

### R2：多场景注册与治理仍处于首版

当前已有统一的 `scenario registry`，但生产配置仍以首个 PoC 场景为主。

风险：

- 场景增多后可能出现调用方直接依赖具体场景常量
- 路由、Service 和 Schema 可能按场景复制扩散
- 场景的默认值、版本和可用状态缺少统一治理

触发条件：新增第二个决策场景，或需要按场景启停、版本切换和权限管理时，应补强多场景注册治理。

### R3：数据 Provider 仍是 PoC 级输入形态

当前数据获取层已有统一 Service 和 `DataPack`，但主要仍依赖 inline 输入。

风险：

- 真实表单、数据库或外部业务系统接入时需要重新明确边界
- 数据来源、权限、更新时间和失败重试尚未形成完整协议
- PoC Provider 不应直接被视为稳定生产数据入口

触发条件：接入真实数据源、跨多个输入来源合并数据，或需要数据新鲜度和权限控制时，应优先补强 Provider 抽象。

### R4：Decision Contract 仍偏向单场景

当前结构化结果已经覆盖 `decision`、`reasoning`、`missing_fields`、`requirement_statuses`、`citations` 和 `debug`，但字段形态仍主要服务于材料核验。

风险：

- 不同业务问题可能各自增加一套 Response Schema
- 规则判断、资料建议和待确认结果的边界可能混在一起
- 前端或 Agent 消费方难以依赖稳定的公共契约

触发条件：出现第二类决策问题，或同一结果需要同时被 HTTP、前端和 Agent 消费时，应抽象通用 Decision Contract。

### R5：规则匹配算法仍偏轻量

当前匹配主要依赖文本归一化、别名和子串包含判断，适合首个固定材料清单场景。

风险：

- 短别名可能造成误命中
- 相似词、包含词和上下文条件可能造成误判
- 复杂制度条款、组合条件和否定表达的适配能力有限

触发条件：出现明显误判或漏判，且无法通过补充别名和固定规则解决时，应升级匹配策略并增加对照评测集。

### R6：规则表达能力尚未覆盖复杂必填语义

规则结构中已经预留 `required` 等字段，但可选项、条件项、替代项和组合项尚未形成完整执行语义。

风险：

- 当前规则容易默认所有要求都是同等必填
- 无法稳定表达“满足任一项”“满足条件后才需要”“附件可替代正文”等规则
- 复杂规则可能被错误简化为普通字符串匹配

触发条件：业务规则出现非全量必填、条件分支或替代材料时，应先补齐领域表达和判定语义，再扩展场景。

### R7：跨场景回归资产仍需扩充

当前已有首个场景的固定样例、应用层回归和 HTTP Smoke，但跨场景、跨数据源和复杂规则的样本还不够丰富。

风险：

- 扩展时容易只验证“当前案例能跑通”
- 规则、数据和结果契约的回归覆盖不足
- 检索证据变化可能影响最终决策，但缺少稳定对照样本

触发条件：新增第二个场景或接入真实 Provider 时，应同步增加正常、缺失、证据不足、误命中和失败分支的固定回归样例。

## 3. 优先级建议

| 优先级 | 风险 | 建议处理时机 |
| --- | --- | --- |
| P1 | 规则外置化、多场景 Registry、Provider 多来源、Decision Contract | 对应扩展条件出现时优先处理 |
| P2 | 匹配算法、规则充分性、`required / optional / conditional` 语义 | 第二类规则问题接入时处理 |
| P3 | 跨场景评测集、Debug/Trace 和回归资产扩充 | 每次扩展场景或数据源时持续补强 |

## 4. 什么时候应暂停主线处理风险

出现以下任一情况时，不建议继续叠加场景代码，应先处理对应技术债：

- 新增第二个规则审核场景后出现明显复制粘贴
- 路由、Service 或 Schema 开始按场景重复实现
- 需要接入真实数据源而不是 inline 输入
- 出现无法通过规则配置或别名修正的明显误判
- 同一结果需要被多个外部消费者使用，但字段语义不一致
- 规则出现条件项、替代项或组合项，而当前模型无法准确表达

## 5. 扩展时必须保持的边界

- 继续遵守 `agent.md` 和 `ARCHITECTURE.md` 的依赖方向
- 具体数据库、LLM、文件和外部系统实现仍由 Infrastructure 提供
- 具体适配器仍由 Composition Root 组装
- 事实性结果必须保留文档、版本、章节、页码和原文来源
- 没有充分证据时返回资料不足，不用模型常识补齐企业事实
- 招标书适用性、资料适用性和最终内容仍需人工确认
- 不因为扩展风险而提前建设通用银行模板或完整多场景 Agent

## 6. 当前结论

这些问题是 Phase 3 的扩展风险和后续技术债，不是当前架构故障，也不阻塞 Phase 2 的完成。

后续采用“触发条件驱动治理”：没有出现对应扩展条件时保持当前 PoC 简洁；一旦出现多场景、多数据源、复杂规则或跨消费者复用需求，就先处理相应风险，再继续扩大业务范围。
