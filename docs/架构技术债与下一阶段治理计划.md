# 架构技术债与下一阶段治理计划

本文档记录的架构边界技术债已完成治理收口。它不是当前 D5 的验收记录，也不改变 `ARCHITECTURE.md` 的有效性。

## 1. 当前判断

当前代码主体已经按照 `ARCHITECTURE.md` 的模块划分和依赖方向建设：

- `Online / Knowledge / Ingestion` 已分离
- `Interfaces / Infrastructure / Composition` 已形成独立边界
- Domain 没有直接依赖 HTTP、数据库或 LLM 框架
- Repository 和外部服务通过能力接口接入

本轮治理已完成边界收紧和架构测试补强，不改变 D5 的业务闭环，只收敛接口层与基础设施层的依赖关系。

## 2. 已治理事项

### 2.1 HTTP 接口层少量引用 Infrastructure

本项治理针对的历史问题是部分 HTTP 路由或依赖注入模块直接引用 Infrastructure 中的具体类型或数据库健康检查：

- `app/interfaces/http/routes/retrieval.py`
- `app/interfaces/http/routes/policy_decision.py`
- `app/interfaces/http/routes/policy_pipeline.py`
- `app/interfaces/http/dependencies.py`

这些引用当时主要用于异常映射、文件上传依赖和会话装配，功能上可工作，但扩大了接口层对具体基础设施的感知范围。

本轮已完成：

- HTTP 路由不再直接导入 `app.infrastructure`
- 应用启动入口和检索评测脚本不再直接导入 `app.infrastructure`
- 数据库缺表错误由 Infrastructure 转换为共享应用异常，再由 HTTP 映射为 `503`
- 上传暂存能力通过 `UploadStoragePort` 暴露，HTTP 不再引用 `PolicyUploadService`

### 2.2 Composition Root 与 HTTP 依赖提供器边界不够纯

历史上 HTTP 依赖提供器负责获取 `ApplicationContainer`，并在类型标注或会话提供过程中接触部分基础设施类型；本轮已将这些具体选择收敛到 Composition Root。

本轮已完成：

- HTTP 层只依赖应用服务契约和依赖提供器
- 数据库会话生命周期已移动到 Composition Root
- 具体 Repository、文件服务、数据库会话和外部适配器仍由 Composition Root 选择和组装
- 应用启动检查与检索评测通过 Composition Root 获取基础设施能力
- HTTP 依赖提供器只暴露 Application 用例和应用端口

### 2.3 架构边界测试覆盖不足

本轮已补充基于 AST 的架构边界测试，覆盖：

- HTTP 包是否直接导入 `app.infrastructure`
- 应用入口和评测脚本是否直接导入 `app.infrastructure`
- Application Modules 是否直接导入 `app.infrastructure`
- Ports 是否反向依赖 `app.infrastructure`
- 具体适配器是否在 Composition Root 之外实例化
- 所有 Domain 是否依赖 HTTP、数据库或外部框架

这些测试覆盖本轮明确治理的边界，并为同类依赖回漏提供自动化拦截。

## 3. 治理结果与 D5 的关系

这些问题此前不改变 D5 的核心结果：

- D5 规则、数据和结果链路仍通过 Application Service 编排
- 现有 Domain 规则不依赖 HTTP 或具体基础设施
- HTTP Smoke 验证了接口到应用服务的基本调用链路
- 接口层的 Infrastructure 引用没有改变当时的业务结果

因此本轮处理结论是：

> **D5 已完成验收，架构技术债已完成治理，不再阻塞后续阶段交接。**
