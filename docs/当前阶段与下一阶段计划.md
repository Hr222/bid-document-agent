# 当前阶段与下一阶段计划

## 1. 当前阶段结论

本项目整体上是一个 **Agent 工程项目**，目标是沉淀作者过往真实业务工作中的能力、规则、案例、数据协同方式和文档产出流程。

当前这次阶段性交付，并不代表项目本身只是 RAG 或知识库项目。更准确地说：

- 项目整体是 agent 工程
- 当前先打通的是其中一段最关键、最容易形成底座能力的子链路
- 这条子链路就是“知识获取 + 检索 + 最小回答 + 调试”

因此，当前阶段可以表述为：

**Agent Phase 1 中的 Retrieval-grounded MVP 已完成。**

## 2. 当前已经做到的事情

### 2.1 知识获取与入库底座已打通

- 支持 `docx / pdf` 单文件入库
- 文档可经过解析、清洗、章节拆分、chunk 切分
- chunk 可生成 embedding 并写入 PostgreSQL/pgvector
- 当前入库主状态允许 `draft` 数据进入知识库

### 2.2 检索子链路已打通

- 已提供统一检索 API：
  - `POST /api/v1/kb/retrieval/search`
  - `POST /api/v1/kb/retrieval/ask`
- 检索和问答共用同一套 retrieval chain
- 查询向量与文档向量都继续使用 **Gitee embedding**
- 检索方式为：
  - `pgvector + cosine distance`
  - 当前是**精确向量检索**
  - 当前不是 ANN，不是 HNSW

### 2.3 最小 GLM 回答已打通

- `ask` 会先走 `search`
- 只有命中检索结果后才会调用 GLM
- 无命中时直接返回“未在知识库中找到足够依据”
- 当前上下文控制规则：
  - 最多取前 `6` 条 hit
  - 每条最多取 `500` 字符

### 2.4 调试工作台已具备

- 现有单页工作台保留了“单文件入库工作台”
- 已新增“检索调试台”
- 可直接看到：
  - answer
  - citations
  - retrieval hits
  - 每条 chunk 的 score / version / page / text

### 2.5 自动验证已通过

- `tests/test_retrieval.py` 已通过
- `tests/test_policy_ingestion.py` 已通过
- `frontend` 构建已通过

## 3. 当前阶段的准确定位

当前阶段解决的是：

- agent 如何先获得稳定的知识获取底座
- agent 如何先获得可复用的检索子能力
- agent 如何先获得最小可验证的基于证据回答能力
- 前后端如何对这段能力进行调试和验证

当前还没有解决的是：

- 多数据源联动型 agent
- 规则与数据库联动计算型 agent
- 基于历史案例和模板的复杂文档生成 agent
- 多步骤编排
- 多轮状态
- 自动评测体系
- 完整 tracing / audit

所以，当前不是“项目完成了一个 RAG 系统”，而是：

**项目完成了 Agent 底座中的第一段关键能力闭环。**

## 4. 当前技术边界

以下内容是当前阶段的明确边界：

- embedding 供应商未切换
  - 文档向量和查询向量仍使用 Gitee
- GLM 当前只承担回答模型
  - 当前不承担 embedding
- 检索仍是单路向量召回
  - 没有接 BM25 / 关键词召回
- 当前不是 HNSW
  - 仍是精确余弦检索
- 当前不是完整 agent 编排框架
  - 未接入 LangChain / LangGraph
- OCR 不作为当前阶段的重点演进方向

## 5. 下一阶段建议定位

下一阶段不应只继续围绕“问答”展开，而应开始往更接近真实业务 agent 的方向演化。

建议定义为：

**Agent Phase 2: 检索增强 + 规则联动底座**

## 6. 下一阶段应该做什么

### 6.1 向量检索升级为 HNSW

目的：

- 提升 chunk 规模上来后的检索性能
- 为后续更大规模知识底座做准备

注意：

- HNSW 是向量索引加速方案
- 它不是混合检索本身

### 6.2 从单路向量检索升级为混合检索

建议演化为：

- 向量召回
- 关键词 / BM25 召回
- 结果融合

这样做更适合制度、规则、条款、编号类内容的稳定召回。

### 6.3 开始抽象“规则获取 + 数据获取 + 结果生成”链路

这是从“检索型子能力”走向“业务 agent”最关键的一步。

建议开始把典型场景拆出来，例如：

- 从知识库获取财务计算规则
- 根据业务规则确定应该查哪些数据库字段
- 拉取业务数据
- 按规则计算
- 生成审核表单或辅助判断

### 6.4 建立历史案例与模板能力

针对投标书生成这类场景，下一阶段应开始整理：

- 可复用的模板
- 历史案例
- 章节结构
- 可替换变量
- 可引用知识来源

### 6.5 建立最小评测集

建议至少分别建立三类验证样本：

- 检索类
- 规则计算类
- 文档生成类

## 7. LangChain / LangGraph 适合什么时候接入

当前阶段还不必急于引入 LangGraph。

更合理的顺序是：

1. 先把检索增强做稳
2. 先把规则与数据联动链路抽象清楚
3. 再决定是否引入编排框架

建议判断标准如下。

### 适合先接 LangChain 的时机

当我们需要：

- retriever 抽象
- prompt 组装模板化
- retrieval / rerank / llm 串联
- 更快试验不同 pipeline

### 适合再进 LangGraph 的时机

当我们需要：

- 多步骤能力编排
- 条件分支
- 失败回退
- 规则获取和数据库获取的动态决策
- 多轮状态
- 可恢复工作流

## 8. 推荐的阶段演化路线

### 当前

**Agent Phase 1 / Retrieval-grounded MVP**

- 单文件入库
- Gitee embedding
- pgvector 精确余弦检索
- 最小 GLM 回答
- 前端检索调试台

### 下一阶段

**Agent Phase 2 / Retrieval + Rule/Data Grounding**

- HNSW
- 混合检索
- rerank
- 规则获取抽象
- 数据获取抽象
- 结果生成链路

### 再下一阶段

**Agent Phase 3 / Orchestrated Business Agents**

- LangChain / LangGraph 视复杂度接入
- 多步骤编排
- 案例与模板协同
- 审核表单 agent
- 投标书生成 agent
- tracing / audit / evaluation

## 9. 当前结论

当前这次交付可以视为：

**阶段性成功，项目已经完成了 Agent 工程中的第一段关键底座：以知识获取、检索和最小回答为核心的能力闭环。**

下一步最重要的不是把项目继续收缩成“只是 RAG”，而是开始往真实业务 agent 能力演化：

- 检索增强
- 规则联动
- 数据联动
- 案例与模板联动
- 编排能力演化
