# 当前阶段与下一阶段计划

## 1. 当前阶段结论

本项目整体上是一个 **Agent 工程项目**，目标不是只做一个问答式 RAG，而是逐步沉淀真实业务工作中会反复依赖的：

- 知识获取
- 检索与证据定位
- 规则理解
- 数据联动
- 结果生成

当前阶段的准确表述应当是：

- **Agent Phase 1 / Retrieval-grounded MVP：已完成**
- **Agent Phase 2 / Retrieval + Rule/Data Grounding：进行中**
- **当前执行阶段：Milestone D / Step D3**

也就是说，项目已经不再停留在“先证明检索闭环能不能跑通”，而是进入“基于已完成的检索底座，开始做最小规则 / 数据 / 结果闭环”的阶段。

## 2. 当前已经完成的能力

### 2.1 知识获取与入库底座已打通

- 支持 `docx / pdf` 单文件入库
- 文档可经过解析、清洗、章节拆分、chunk 切分
- chunk 可生成 embedding 并写入 `PostgreSQL/pgvector`
- 当前允许 `draft` 状态数据进入知识库

### 2.2 基础检索与最小问答已打通

- 已提供统一检索 API：
  - `POST /api/v1/kb/retrieval/search`
  - `POST /api/v1/kb/retrieval/ask`
- `search` 与 `ask` 共用同一条 retrieval pipeline
- 查询向量与文档向量继续使用 `Gitee embedding`
- `ask` 先走 `search`，只有命中结果后才调用 GLM 生成回答

### 2.3 当前已具备的检索增强能力

- 已接入向量召回 + 关键词召回双路 pipeline
- 已完成第一版结果融合与去重
- 已加入当前可用的启发式 rerank
- 已补齐检索 debug 信息，能够解释：
  - 命中了什么
  - 来自哪一路召回
  - 分数是如何拆解的
- 已落地最小 baseline 脚本与评测集
- 已完成 `Milestone B / C` 暴露问题的第一轮收口与复验

### 2.4 当前已开始具备的 D1 闭环能力

- 已选定首个真实业务 PoC 场景：
  - `委托评估机构申请材料核验`
- 已新增独立 PoC 路由：
  - `POST /api/v1/kb/policy-decisions/court-evaluation-materials/review`
- 已具备一版规则驱动清单校验能力
- 已具备最小 inline data provider
- 已具备结构化 decision 输出：
  - `pass / fail / insufficient_evidence`
  - `missing_fields / citations / debug`
- 已完成单元测试与真实 public 数据 smoke

## 3. 当前阶段的准确定位

在 `Milestone C` 完成后，当前阶段开始从“检索底座工程化”继续往“最小业务闭环”推进。

当前阶段解决的核心问题变为：

1. 系统能否基于现有检索底座稳定拿到可引用的规则依据
2. 在规则依据具备后，系统能否接上最小数据获取与结果生成链路

因此，当前阶段的核心工作不再是继续围绕已有能力做小样本微调，而是进入：

**Milestone D：规则获取 + 数据获取 + 结果生成 PoC**

当前工作下标：

- **Milestone D / Step D3：已完成首版落地**
- 当前判断：`D1` 已完成首个可运行 PoC，`D2` 已完成规则获取层首版抽象，`D3` 已完成数据获取层首版抽象，下一步应继续推进 `D4`

## 4. 当前技术边界

以下内容仍然是当前阶段的明确边界：

- embedding 供应商暂不切换
  - 文档向量与查询向量继续使用 Gitee
- GLM 当前仍主要承担回答模型角色
- 当前检索主链路已是：
  - 向量召回
  - 关键词召回
  - 融合
  - rerank
- OCR 仍不是当前阶段优先推进重点
- 当前仍未进入完整业务 Agent 编排阶段
  - 暂未接入 `LangChain / LangGraph`
- 当前仍不直接接正式外部业务系统
  - 优先使用 mock / 静态样例 / 手工补充字段完成 PoC

- **Agent Phase 2 / Milestone C：已完成**
- **Agent Phase 2 / Milestone D：当前正在推进**

这意味着项目已经从“先证明 RAG 闭环可行”进入到“基于检索底座，开始做最小业务判断 PoC”的阶段。

## 6. 当前阶段调整后的定位

现阶段更适合明确成：

**Agent Phase 2：检索工程化增强 + 规则/数据闭环准备**

这一阶段的目标不是立刻做复杂业务 Agent，而是先把后续 Agent 真实依赖的检索、规则、数据与结果输出链路做成最小可验证结构。

## 7. Phase 2 的里程碑拆分

### 7.1 Milestone A：检索底座增强

这是 Phase 2 的第一优先级，也是后续所有演化的基础。

本里程碑的目标：

- 建立最小评测集，并把评测前置
- 抽象当前 retrieval chain，使其不再绑定单一路径
- 为调试台补充更适合检索实验的中间信息
- 为后续混合检索、rerank、LangChain retriever 接口做结构准备

建议本里程碑完成的内容：

- 检索类样本集
- 检索效果基线
- 多阶段检索响应结构
- 可扩展的 retrieval pipeline 抽象

当前状态：

- 已完成检索链路抽象
- 已完成调试能力增强
- 已补齐最小评测集与 baseline 骨架

因此，`Milestone A` 当前可以视为已完成收尾，下一步进入 `Milestone B`。

对应工作记录文档已切换为：

- `docs/第二阶段-MilestoneB工作计划.md`

### 7.2 Milestone B：混合检索与 rerank

在当前制度、规则、条款、编号类内容场景下，混合检索通常比单纯更换索引更直接影响召回质量。

建议演化为：

- 向量召回
- 关键词 / BM25 召回
- 结果融合
- rerank

本里程碑的核心目标不是“加新技术名词”，而是提升：

- 制度条款类问题的稳定召回
- 编号、关键词、专有名词类问题的命中率
- 问答前证据质量

### 7.3 Milestone C：HNSW 与检索性能准备

HNSW 在本阶段的重要性主要体现在性能和规模准备，而不是直接替代混合检索。

这是上一阶段的主任务，当前已完成。

本里程碑目标：

- 为 chunk 数量增长后的向量检索做性能准备
- 为更大规模制度知识库保留可扩展空间
- 在不破坏当前混合检索可用性的前提下，把检索策略做成可切换、可验证、可比较

注意：

- HNSW 是向量索引加速方案
- 它不是混合检索本身
- Milestone C 的重点不只是“加索引”，而是“把 exact 与 HNSW 的工程结构、配置、验证方式补齐”

当前工作文档为：

- 当前仅在阶段总览中保留完成状态，不再单独维护 C 阶段计划文档

### 7.4 Milestone D：规则获取 + 数据获取 + 结果生成 PoC

这一阶段才开始从“检索子能力”进一步走向“业务 Agent 子链路”。

建议仍然先选一个最小真实业务场景做 PoC，例如：

- 获取一类规则
- 判断需要哪些业务字段
- 拉取最小业务数据
- 按规则计算
- 输出结论或辅助判断结果

当前工作文档为：

- `docs/第二阶段-MilestoneD工作计划.md`

当前状态：

- 已完成 `Step D1`
- 已完成 `Step D2`
- 已完成 `Step D3` 首版落地
- 下一步应优先进入 `Step D4`，继续把结果生成链路抽象做完整

### 7.5 Milestone E：历史案例与模板能力准备

这部分很重要，但更适合作为 Phase 2 后段的准备项，而不是当前最先推进的主任务。

针对投标书生成类场景，可逐步整理：

- 可复用模板
- 历史案例
- 章节结构
- 可替换变量
- 可引用知识来源

这部分完成后，后续文档生成型 agent 会更容易落地。

## 8. 当前建议优先级

建议按以下顺序推进：

1. `P0` Milestone D：继续完成 D3 收口并进入 D4
2. `P1` 增补 D 阶段 PoC 回归资产与第二类场景验证
3. `P2` 历史案例 / 模板能力整理

其中当前最重要的判断原则是：

- 先基于已经完成的检索底座跑通一条最小规则 / 数据 / 结果闭环
- 再补齐更稳的规则获取抽象与回归验证资产
- 最后再继续进入更复杂的业务 Agent 抽象

## 9. 下一阶段目标：LangChain / LangGraph 接入

当 Phase 2 的检索与规则/数据基础准备到一定程度后，项目就适合自然进入下一阶段：

**Agent Phase 3 / Orchestrated Business Agents**

这一阶段的重点不再是继续堆叠单点检索能力，而是把已经沉淀下来的能力组织成可编排的业务 Agent。

### 9.1 适合先接入 LangChain 的时机

当我们具备以下基础时，可优先接入 `LangChain`：

- retrieval pipeline 已经被抽象清楚
- 混合检索 / rerank / HNSW 已有稳定实验方式
- prompt 组装开始需要模板化
- 规则获取、数据获取、结果生成开始需要统一 tool / chain 组织

### 9.2 适合再接入 LangGraph 的时机

当我们真正开始遇到以下问题时，再接入 `LangGraph` 更合理：

- 多步骤编排
- 条件分支
- 失败回退
- 动态决策到底取规则、取数据还是继续追问
- 多轮状态
- 可恢复工作流
