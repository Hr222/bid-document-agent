# 当前阶段与下一阶段计划

本文档记录项目当前已经完成的阶段、当前真实状态，以及下一阶段的开发边界。后续开发以仓库中的实际代码和以下架构基准为准：

- `ARCHITECTURE.md`
- `README.md`
- `agent.md`
- `docs/第三阶段工作计划.md`

## 1. 当前总判断

当前项目已经完成 `Agent Phase 1` 和 `Agent Phase 2`，下一步进入 `Agent Phase 3`。

当前阶段划分如下：

| 阶段 | 定位 | 状态 |
|---|---|---|
| Agent Phase 1 | 资料知识底座建设 | 已完成 |
| Agent Phase 2 | 检索基础设施与规则/数据能力建设 | 已完成 |
| Agent Phase 3 | 招标书驱动的投标书应用 | 当前阶段 |

项目不再以“继续扩展底层检索能力”作为当前主线，而是开始把已经完成的知识底座接入具体应用流程。

## 2. Agent Phase 1：资料知识底座建设

第一阶段主要解决：

> 资料能否进入系统，并成为可读取、可检索的知识。

已完成的主要内容：

- 文件接入和资料准入校验
- DOC / DOCX / PDF / 图片解析
- OCR 识别与 OCR 结果回填
- 文本清洗
- 章节拆分
- 文档块和 Chunk 切分
- Embedding 生成
- PostgreSQL / pgvector 持久化
- 最小检索和 RAG 问答验证

第一阶段的本质是：

**资料清洗、解析、OCR、入库，并验证知识底座具备基本检索能力。**

第一阶段不负责理解具体银行业务，也不负责生成投标书。

## 3. Agent Phase 2：检索基础设施与规则/数据能力建设

第二阶段主要解决：

> 知识底座能否稳定、可解释、可扩展地支撑后续应用。

### 3.1 Milestone A：检索底座抽象与评测

- 抽象 retrieval pipeline
- 建立最小检索评测集
- 增加检索过程调试信息
- 建立 baseline 和可重复验证方式

### 3.2 Milestone B：混合检索与 rerank

- 向量召回
- 关键词召回
- 结果融合和去重
- 启发式 rerank
- 召回来源和分数拆解

### 3.3 Milestone C：HNSW 与性能准备

- Exact / HNSW 向量策略切换
- HNSW 索引脚本
- `ef_search` 等参数配置
- 数据库集成测试
- 检索性能和规模扩展准备

### 3.4 Milestone D：规则、数据与结果 PoC

- D1：确定“委托评估机构申请材料核验”场景
- D2：抽象 `RulePack` 和规则获取服务
- D3：抽象 `DataPack`、数据 Provider 和来源追踪
- D4：完成规则、数据、结果的结构化编排
- D5：固定 PoC 样例、应用层回归和 HTTP Smoke

当前 D5 代码、自动化验证和架构技术债治理已经完成：

- 固定 4 类 PoC Case
- 应用层 fixture 驱动回归测试
- HTTP API Smoke 测试
- 全量后端测试：`85 passed`
- `ruff check app tests` 通过
- `python -m compileall -q app tests` 通过
- `frontend/npm.cmd run build` 通过
- HTTP / Application / Ports 边界治理和 AST 架构回归测试已完成
- 机测和人测已由项目负责人确认，D5 正式验收完成

### 3.5 Milestone E：知识库收尾

Milestone E 是第二阶段最后一个工作阶段，定位不是模板化或历史案例工程化，而是：

> **保证现有资料干净、可查、可追溯，能够作为应用端后续使用的可靠知识底座。**

第二阶段最终沉淀的内容是：

```text
资料知识底座
  -> 混合检索与 rerank
  -> HNSW 性能准备
  -> 规则 / 数据 / 结果 PoC
  -> 干净、可查、可追溯的知识库
```

当前资料已完成技术盘点，具备解析结果、版本信息、章节结构、检索切块和来源追踪能力。

Milestone E 已完成收尾：

- 新增只读知识库审计能力和可重复执行脚本，覆盖资料数量、状态、解析结果和质量问题清单
- 检索命中、RAG 引用和规则决策引用均保留源文件路径与文件名，可继续回到文档、版本、章节、页码和原文片段
- 审计发现的历史源文件不可访问、章节标识缺失等问题已记录，不做未经业务确认的破坏性清理
- 后端全量测试、静态检查、编译检查和前端构建均通过

因此，Agent Phase 2 已完成，当前项目进入 Agent Phase 3。

这些资料可作为后续应用端使用的企业事实和证明依据，但具体适用范围仍由业务人员结合招标书人工确认。文档、测试和阶段计划不展开记录真实资料的数量、类别、名称或内容。

第二阶段明确不负责：

- 通用投标书模板
- 跨银行格式统一
- 历史标书模板库
- 自动生成完整投标书
- 具体银行业务规则理解

## 4. Agent Phase 3：招标书驱动的投标书应用

第三阶段的核心目标是：

> **理解一份具体招标书，生成符合这份招标书要求的投标书格式文件，使用知识库填写内容，最后由人工确认。**

核心链路为：

```text
招标书
  -> 招标书理解
  -> 生成该招标书专属的投标书格式结构
  -> 内容填写
  -> 人工确认
```

招标书是当前项目的格式和要求来源，不预先建立适用于所有银行的统一模板。

## 5. Phase 3 的分段推进方式

第三阶段不一次性建设完整应用，而是后端和前端按能力逐段匹配，每完成一段就进行测试和人工验证。

### F1：招标书输入与应用边界

目标：让用户能够向系统提交一份具体招标书，并获得明确的解析任务。

后端：

- 定义招标书应用用例
- 复用现有文件接入和解析能力
- 定义招标书解析请求和结果契约
- 保留原始文件和来源追踪

前端：

- 最小文件上传界面
- 解析状态展示
- 失败和待处理状态展示

这一阶段不提前建设完整工作台。

### F2：招标书理解

目标：使用大模型 Prompt 从具体招标书中提取结构化要求。

重点提取：

- 章节结构和章节顺序
- 必填章节
- 资格要求
- 评分要求
- 证明材料要求
- 附件要求
- 格式和提交约束
- 每条要求对应的原文引用
- 无法确定的内容和置信提示

实现约束：

- Prompt 编排放在应用层
- LLM 客户端仍属于基础设施适配器
- 模型输出必须经过结构化 Schema 校验
- 不允许把模型自由文本直接当作业务事实
- 解析结果必须保留招标书原文引用

前端：

- 展示招标书章节和要求
- 展示原文引用
- 展示待确认或无法判断的条目

### F3：生成招标书指定的投标书格式结构

目标：根据当前招标书的要求，生成当前项目专属的投标书结构文件或结构化文件草稿。

重点是：

- 按招标书要求排列章节
- 生成招标书要求的标题、目录和应答位置
- 保留附件和证明材料位置
- 标记尚未填写的内容区域

这里生成的是“当前招标书的应答结构”，不是通用模板系统。

前端：

- 展示投标书章节结构
- 展示章节顺序和必填状态
- 支持进入某个章节查看待填写内容

### F4：内容填写

目标：使用 Phase 1 和 Phase 2 已经建设的知识能力，为当前投标书结构填充有依据的内容。

可复用能力包括：

- 文档和资料解析结果
- 混合检索、rerank 和 HNSW
- RAG 引用能力
- 规则获取和数据获取能力
- `RulePack`、`DataPack` 和结构化结果
- 来源、版本、章节、页码和原文追踪

填写约束：

- 有资料依据才能填写事实性内容
- 没有资料依据必须明确标记缺失
- 不根据模型常识编造企业事实
- 不擅自改变招标书要求的章节结构
- 不自动将历史内容判断为当前项目适用内容

前端：

- 展示候选内容
- 展示引用资料
- 展示资料版本和来源
- 展示待补内容

### F5：人工确认

目标：由业务人员确认招标书理解结果、资料适用性和最终填写内容。

系统负责：

- 展示待确认项
- 保留原始资料引用
- 保留修改前后内容
- 输出最终待确认文件

系统暂不负责：

- 替代业务人员判断
- 自动决定内容是否适合当前项目
- 自动提交最终投标文件

前端：

- 提供确认、修改和标记待补能力
- 清晰区分系统建议、原始资料和人工修改

## 6. Phase 3 的前端策略

现有前端主要承担入库和检索调试作用，第三阶段开始正式建设应用端前端，但仍然采用小步方式：

```text
后端完成一段应用能力
  -> 确定 HTTP Schema
  -> 前端实现对应最小页面
  -> pytest、前端构建和人工测试
  -> 再进入下一段
```

不提前建设完整投标工作台，不提前确定所有页面、字段和交互。

前端必须通过 HTTP API 使用后端能力，不直接访问数据库、Repository 或内部领域对象。

## 7. Phase 3 不做什么

当前阶段暂不做：

- 通用银行投标模板
- 跨银行格式抽象
- 没有真实招标书依据的自动写作
- 未经人工确认的正式投标文件提交
- 一次性建设完整多场景 Agent
- 一开始就引入 LangGraph 复杂编排
- 直接接入正式外部业务系统

LangChain / LangGraph 是否接入，取决于后续是否真正出现多步骤、条件分支、失败回退和可恢复状态等编排需求。

## 8. 架构与质量约束

第三阶段继续严格遵守当前架构：

- HTTP 接口层只负责 Schema、Assembler、依赖注入和异常映射
- 应用编排放在对应 Application Service
- 规则和业务契约放在 Domain / Ports
- LLM、文件、OCR、数据库等具体实现放在 Infrastructure
- 具体依赖统一由 Composition Root 组装
- Domain 不依赖 HTTP、数据库、LLM 框架或具体外部适配器
- 前端只依赖稳定 HTTP API
- 重要复杂逻辑按 `agent.md` 添加必要注释
- 每段能力都必须补充自动化测试，外部模型测试使用稳定替身或固定输出

## 9. 下一步执行顺序

第三阶段第一步不是写投标正文，而是：

1. 选择一份真实招标书作为第一个应用样例。
2. 定义招标书理解的结构化输入输出契约。
3. 设计第一版 Prompt 和模型输出校验。
4. 通过后端 API 暴露招标书理解结果。
5. 前端只实现上传、解析状态和结构化结果展示。

完成 F1-F2 并经过人工确认后，再进入 F3 的投标书格式结构生成。

## 10. 当前一句话总结

**第一阶段建设资料知识底座，第二阶段建设检索与规则/数据基础设施，第三阶段开始建设招标书驱动的投标书应用，并由前后端按能力逐段推进。**
