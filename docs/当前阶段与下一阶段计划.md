# 当前阶段与下一阶段计划

## 1. 当前阶段结论

本项目整体上是一个 **Agent 工程项目**，目标是沉淀作者过往真实业务工作中的能力、规则、案例、数据协同方式和文档产出流程。

当前这次阶段性交付，并不代表项目本身只是 RAG 或知识库项目。更准确地说：

- 项目整体是 agent 工程
- 当前先打通的是其中一段最关键、最容易形成底座能力的子链路
- 这条子链路就是“知识获取 + 检索 + 最小回答 + 调试”

因此，当前阶段可以表述为：

**Agent Phase 1 中的 Retrieval-grounded MVP 已完成。**

## 2. 当前已经做到的事情

### 2.1 知识获取与入库底座已打通

- 支持 `docx / pdf` 单文件入库
- 文档可经过解析、清洗、章节拆分、chunk 切分
- chunk 可生成 embedding 并写入 PostgreSQL/pgvector
- 当前入库主状态允许 `draft` 数据进入知识库

### 2.2 检索子链路已打通

- 已提供统一检索 API：
  - `POST /api/v1/kb/retrieval/search`
  - `POST /api/v1/kb/retrieval/ask`
- 检索和问答共用同一套 retrieval chain
- 查询向量与文档向量都继续使用 **Gitee embedding**
- 检索方式为：
  - `pgvector + cosine distance`
  - 当前是**精确向量检索**
  - 当前不是 ANN，不是 HNSW

### 2.3 最小 GLM 回答已打通

- `ask` 会先走 `search`
- 只有命中检索结果后才会调用 GLM
- 无命中时直接返回“未在知识库中找到足够依据”
- 当前上下文控制规则：
  - 最多取前 `6` 条 hit
  - 每条最多取 `500` 字符

### 2.4 调试工作台已具备

- 现有单页工作台保留了“单文件入库工作台”
- 已新增“检索调试台”
- 可直接看到：
  - answer
  - citations
  - retrieval hits
  - 每条 chunk 的 score / version / page / text

### 2.5 自动验证已通过

- `tests/test_retrieval.py` 已通过
- `tests/test_policy_ingestion.py` 已通过
- `frontend` 构建已通过

## 3. 当前阶段的准确定位

当前阶段解决的是：

- agent 如何先获得稳定的知识获取底座
- agent 如何先获得可复用的检索子能力
- agent 如何先获得最小可验证的基于证据回答能力
- 前后端如何对这段能力进行调试和验证

当前还没有解决的是：

- 多数据源联动型 agent
- 规则与数据库联动计算型 agent
- 基于历史案例和模板的复杂文档生成 agent
- 多步骤编排
- 多轮状态
- 自动评测体系
- 完整 tracing / audit

所以，当前不是“项目完成了一个 RAG 系统”，而是：

**项目完成了 Agent 底座中的第一段关键能力闭环。**

## 4. 当前技术边界

以下内容是当前阶段的明确边界：

- embedding 供应商未切换
  - 文档向量和查询向量仍使用 Gitee
- GLM 当前只承担回答模型
  - 当前不承担 embedding
- 检索仍是单路向量召回
  - 没有接 BM25 / 关键词召回
- 当前不是 HNSW
  - 仍是精确余弦检索
- 当前不是完整 agent 编排框架
  - 未接入 LangChain / LangGraph
- OCR 不作为当前阶段的重点演进方向

## 5. 当前推进状态

基于当前仓库实际完成情况，项目状态不再只是“准备进入下一阶段”，而是可以明确表述为：

- **Agent Phase 1 / Retrieval-grounded MVP：已完成**
- **Agent Phase 2 / Retrieval + Rule/Data Grounding：已启动，按里程碑推进**
- **Agent Phase 3 / Orchestrated Business Agents：作为下一阶段目标保留**

这意味着项目已经从“先证明 RAG 闭环可行”进入到“把检索能力继续工程化，并为业务 agent 做底座准备”的阶段。

## 6. 当前阶段调整后的定位

现阶段不再适合继续把工作目标表述成一组并列的功能设想，而更适合明确成：

**Agent Phase 2: 检索增强 + 规则联动底座（按里程碑推进）**

这个阶段的目标不是立刻做一个很大的 agent，而是先把后续 agent 真正依赖的底层能力做稳、做清楚、做可验证。

## 7. Phase 2 的里程碑拆分

### 7.1 Milestone A：检索底座增强

这是 Phase 2 的第一优先级，也是后续所有演化的基础。

本里程碑的目标：

- 建立最小评测集，并把评测前置
- 抽象当前 retrieval chain，使其不再绑定单一路径
- 为调试台补充更适合检索实验的中间信息
- 为后续混合检索、rerank、LangChain retriever 接口做结构准备

建议本里程碑完成的内容：

- 检索类样本集
- 检索效果基线
- 多阶段检索响应结构
- 可扩展的 retrieval pipeline 抽象

### 7.2 Milestone B：混合检索与 rerank

在当前制度、规则、条款、编号类内容场景下，混合检索通常比单纯更换索引更直接影响召回质量。

建议演化为：

- 向量召回
- 关键词 / BM25 召回
- 结果融合
- rerank

本里程碑的核心目标不是“加新技术名词”，而是提升：

- 制度条款类问题的稳定召回
- 编号、关键词、专有名词类问题的命中率
- 问答前证据质量

### 7.3 Milestone C：HNSW 与检索性能准备

HNSW 在本阶段的重要性主要体现在性能和规模准备，而不是直接替代混合检索。

本里程碑目标：

- 为 chunk 数量增长后的向量检索做性能准备
- 为更大规模知识底座保留可扩展空间

注意：

- HNSW 是向量索引加速方案
- 它不是混合检索本身
- 在优先级上，应放在检索评测和混合检索结构清楚之后推进

### 7.4 Milestone D：规则获取 + 数据获取 + 结果生成 PoC

这是项目从“检索子能力”走向“业务 agent 子链路”的关键一步。

建议不要一开始抽象成很大的框架，而是先选一个真实业务场景做最小 PoC，例如：

- 从知识库获取一类财务或审核规则
- 根据规则判断需要查询哪些业务字段
- 拉取最小业务数据
- 按规则计算
- 输出审核结论、表单或辅助判断结果

这个里程碑的价值在于先沉淀出三类边界：

- 规则获取
- 数据获取
- 结果生成

### 7.5 Milestone E：历史案例与模板能力准备

这部分很重要，但更适合作为 Phase 2 后段的准备项，而不是当前最先推进的主任务。

针对投标书生成类场景，可逐步整理：

- 可复用模板
- 历史案例
- 章节结构
- 可替换变量
- 可引用知识来源

这部分完成后，后续文档生成型 agent 会更容易落地。

## 8. 当前建议优先级

建议按以下顺序推进：

1. `P0` 最小评测集 + 检索链路抽象 + 调试能力增强
2. `P1` 混合检索 + rerank
3. `P2` HNSW
4. `P3` 单一规则 / 数据联动 PoC
5. `P4` 历史案例 / 模板能力整理

其中最重要的判断原则是：

- 先解决“能不能验证”
- 再解决“召回好不好”
- 再解决“规模上来后快不快”
- 最后再进入更复杂的业务 agent 抽象

## 9. 下一阶段目标：LangChain / LangGraph 接入

当 Phase 2 里程碑完成到一定程度后，项目就适合自然进入下一阶段。

建议将下一阶段定义为：

**Agent Phase 3 / Orchestrated Business Agents**

这一阶段的重点，不再是继续堆叠单点检索能力，而是开始把已有能力组织成真正可编排的业务 agent。

### 9.1 适合先接入 LangChain 的时机

当我们已经具备以下基础时，可以优先接入 LangChain：

- retrieval pipeline 已经被抽象清楚
- 混合检索 / rerank 已具备稳定实验方式
- prompt 组装开始需要模板化
- 规则获取、数据获取、结果生成开始需要统一 tool / chain 组织

此时 LangChain 更适合作为：

- retriever 抽象层
- prompt 模板层
- retrieval / rerank / llm 串联层
- tool 封装层

### 9.2 适合再进入 LangGraph 的时机

当我们开始真正遇到以下问题时，再接入 LangGraph 会更合理：

- 多步骤能力编排
- 条件分支
- 失败回退
- 动态决策到底取规则、取数据还是继续追问
- 多轮状态
- 可恢复工作流

也就是说，LangGraph 更适合作为“复杂业务 agent 编排层”，而不是当前阶段一开始就引入的基础依赖。

## 11. 当前结论

当前这次交付不应再被表述成“项目只是完成了一个 RAG 系统”，而应更准确地理解为：

**项目已经完成 Agent 工程中的第一段关键底座，并已经进入第二阶段的工程化推进：一边增强检索，一边为规则联动和后续 agent 编排做准备。**

因此，当前最重要的不是继续把项目收缩成“只是问答”，而是按照里程碑有节奏地往下推进：

- 先把检索做成可评测、可调试、可演化的底座
- 再把规则与数据联动链路做成最小业务 PoC
- 然后自然过渡到 LangChain / LangGraph 支撑下的业务 agent 阶段
