# 第三阶段 - 前端改造工作

本文档独立记录第三阶段前端改造工作的范围、实施顺序、实现证据和验收结果。

后端 F1 的范围和进度记录在 [第三阶段- 后端F1工作进度.md](第三阶段-%20后端F1工作进度.md) 中。本文件不修改后端 F1 的计划，也不将前端任务写入后端进度文档。

前端必须通过稳定的 HTTP API 使用后端能力，不直接依赖数据库、Repository、LLM、LangChain 或 LangGraph。

## 1. 阶段定位

第三阶段前端改造分为两个相互衔接但独立验收的方向：

```text
现有知识库调试界面
  -> 知识库正式工作台

现有前端基础壳层
  -> 招标书 Agent F1 最小任务闭环
```

本阶段的视觉参考为 AetherFlow，但 AetherFlow 导出包只作为视觉和交互参考，不直接作为生产代码合并。

## 2. 当前前端状态

### 2.1 当前工程

当前前端工程位于 `frontend/`，使用 React、TypeScript 和 Vite。

现有实现主要集中在以下位置：

```text
frontend/src/
├── App.tsx
├── api/                       # 当前业务请求封装
├── components/                # 上传、检索和结果组件
├── types/                     # 知识库、入库、检索类型
└── styles.css
```

当前已具备：

- 知识文档上传预览
- 文档入库
- 制度文档选项查询
- 知识库检索
- RAG 问答
- 检索引用和调试结果展示

当前主要问题：

- 页面状态、请求流程和业务模块集中在 `App.tsx`
- 请求仍使用原生 `fetch`
- 尚未引入统一 Axios Client
- 尚未引入 React Router 和 TanStack React Query
- 缺少正式知识库列表、文档详情、失败重试和状态恢复页面
- 尚未实现招标书 Agent 的上传、任务状态和骨架结果页面
- 当前页面视觉仍是调试工具风格，未形成统一工作台布局

### 2.2 AetherFlow 参考包评估

`aetherflow.zip` 中包含侧边栏、顶部导航、Agent 列表、知识库页面、Chat 和 Workflow 等视觉原型。

可以复用：

- 深色侧边栏和浅色工作区的布局方向
- 卡片、状态标签、空状态和任务面板的视觉语言
- Agent 列表、任务详情、知识库文档表格的交互结构

不直接复用：

- 硬编码的 Agent、文档、Chunk 和统计数据
- `setTimeout` 模拟任务和聊天结果
- 直接写在页面组件中的假 API 行为
- 与当前工程不一致的 Tailwind、Material Symbols 和外部资源配置
- 原型中的 Workflow 画布和完整 Chat 能力

## 3. 前端架构约束

前端目标结构如下：

```text
frontend/
└── src/
    ├── app/
    │   ├── router.tsx
    │   ├── providers.tsx
    │   └── appConfig.ts
    ├── layouts/
    │   └── AgentWorkspaceLayout.tsx
    ├── features/
    │   ├── knowledge-base/
    │   │   ├── pages/
    │   │   ├── components/
    │   │   ├── hooks/
    │   │   ├── api/
    │   │   └── types.ts
    │   └── agent/
    │       └── tender/
    │           ├── pages/
    │           ├── components/
    │           ├── hooks/
    │           ├── api/
    │           └── types.ts
    ├── services/
    │   └── http/
    │       ├── axiosClient.ts
    │       ├── errorHandler.ts
    │       └── requestTypes.ts
    ├── shared/
    ├── styles/
    └── main.tsx
```

必须遵守：

| 规则 | 要求 |
|---|---|
| 请求入口 | 所有后端请求经过 Axios Client |
| 页面请求 | 页面不直接创建 Axios 请求 |
| 服务端状态 | 使用 TanStack React Query 管理缓存、轮询和重试 |
| 路由 | 使用 React Router，不使用 `ScreenType` 作为长期路由方案 |
| 类型 | 请求类型和响应类型分开定义，保持 TypeScript strict |
| 依赖方向 | 页面 → Hook → Business API → Axios Client → HTTP API |
| 后端边界 | 不访问数据库、Repository、LLM 或 LangChain |
| 模拟数据 | 只允许用于测试替身，不作为完成状态展示 |
| UI 改造 | 渐进式迁移，不一次性重写全部前端 |

## 6. 计划一：知识库前端补充

### 6.1 目标

把现有知识库入库和检索调试界面改造成正式的知识库工作台。知识库解析、清洗、切块、向量化和持久化继续由后端负责，前端只负责业务操作和结果呈现。

### 6.2 实施顺序

#### KB-01：统一请求与类型

- 建立 Axios Client、超时、错误转换和请求取消机制。
- 将现有 `fetch` 请求迁移到 Business API。
- 为上传请求保留上传进度回调。
- 将当前知识库类型拆分为请求类型、响应类型和页面模型。

#### KB-02：知识库工作台

- 实现知识库概览卡片。
- 实现文档列表、搜索和状态筛选。
- 展示文档名称、类型、大小、更新时间、版本和处理状态。
- 增加无知识库、无文档、无结果和接口异常状态。

#### KB-03：上传与处理状态

- 支持拖拽和批量选择文件。
- 展示文件校验、上传、预览和入库阶段。
- 显示待处理、处理中、已完成和失败状态。
- 失败时展示错误信息并支持重新处理。
- 上传流程不将招标书文件写入制度知识库。

#### KB-04：文档详情与来源

- 展示文档元数据和版本信息。
- 展示解析阶段、章节数和 Chunk 数。
- 支持章节 / Chunk 预览。
- 检索结果展示来源文件、版本、章节、页码和原文片段。

#### KB-05：检索与问答

- 区分“只检索”和“RAG 问答”。
- 展示命中结果、相关度、来源和引用。
- 调试信息默认折叠，供开发和排障使用。
- 无有效证据时清晰展示后端拒答原因。

### 6.3 知识库前端依赖的接口

当前可以直接使用：

- `GET /api/v1/kb/overview`
- `GET /api/v1/kb/documents`
- `POST /api/v1/kb/policy-pipeline/preview-upload`
- `POST /api/v1/kb/policy-pipeline/ingest-upload`
- `POST /api/v1/kb/retrieval/search`
- `POST /api/v1/kb/retrieval/ask`
- `POST /api/v1/kb/publication/activate`

正式产品页面还需要在接口契约中明确文档详情、删除、重试、处理状态和 Chunk 预览能力。前端不通过临时拼接内部数据库字段来替代这些接口。

### 6.4 知识库验收标准

- 页面不再使用硬编码文档和 Chunk 数据。
- 可以真实完成上传预览、入库、检索和引用查看。
- 失败文档可以看到原因并重新处理。
- 页面刷新后能够恢复服务端状态。
- 所有请求经过统一 Axios Client。
- 知识库模块可以独立测试和维护。

## 7. 计划二：F1 智能体前端补充

### 7.1 F1 前端范围

```text
上传招标书
  -> 创建 Tender 任务
  -> 查询任务状态
  -> 生成投标书章节骨架
  -> 预览骨架
  -> 下载骨架文件
```

F1 前端暂不实现：

- 完整投标书正文生成
- 知识库内容引用和内容填充
- 多 Agent 协作
- LangGraph Workflow 画布
- 完整 Chat 工作台
- 正式投标文件提交

F1 是否选择知识库以当前后端 F1 范围为准：当前不绑定知识库。前端可以预留位置，但不展示虚假的知识库选择结果，也不向后端发送未定义字段。

### 7.2 实施顺序

#### AG-01：Tender Agent 入口

- 实现招标书文件拖拽上传。
- 支持文件类型和大小校验。
- 展示原始文件名、大小和来源信息。
- 创建任务后跳转任务详情页。

#### AG-02：任务状态页

- 展示 `pending`、`processing`、`completed`、`failed` 和 `waiting_confirmation`。
- 展示当前处理阶段、进度、开始时间和更新时间。
- 使用 React Query 轮询任务状态。
- 终态自动停止轮询。
- 失败时展示错误信息并支持重试。

#### AG-03：骨架结果页

- 使用 Tree 展示章节层级和顺序。
- 展示必填章节、附件位置和待填写占位符。
- 明确区分“结构信息”和“正文内容”。
- 不展示企业事实、知识库内容或模型自由生成的正文。

#### AG-04：下载与结果留痕

- 提供骨架文件下载。
- 展示任务 ID、源文件和生成时间。
- 下载失败时给出可重试提示。
- 前端不自行生成或拼接投标书文件。

### 7.3 建议的 HTTP 使用方式

后端契约确定后，前端按统一 API 模块接入，建议覆盖：

```text
POST /api/v1/agents/tender/tasks
GET  /api/v1/agents/tender/tasks/{task_id}
POST /api/v1/agents/tender/tasks/{task_id}/retry
GET  /api/v1/agents/tender/tasks/{task_id}/skeleton
GET  /api/v1/agents/tender/tasks/{task_id}/download
```

具体路径和字段以 HTTP Schema 为准，前端不提前固化未完成的后端契约。

### 7.4 F1 前端验收标准

- 可以上传真实招标书并创建唯一任务。
- 可以刷新页面后继续查询任务状态。
- 可以展示处理中、完成和失败状态。
- 失败任务可以重试。
- 完成后可以预览章节骨架并下载文件。
- 前端不直接依赖 LLM、LangChain、数据库或 Repository。
- 不把演示数据当成真实 Agent 能力。

## 8. 前后端协作边界

前端改造工作依赖后端 HTTP Schema，但不改变后端 F1 的工作计划。

```text
后端 F1
  -> 提供稳定 HTTP Schema
  -> 前端实现最小页面
  -> 前端构建、自动化测试和人工验证
  -> 记录前端验收证据
```

前端需要后端确认的内容：

- 请求字段和响应字段
- 任务状态枚举
- 错误码和错误信息
- 文件下载方式
- 轮询间隔和终态规则
- 文档处理状态的映射关系

前端不负责：

- 修改后端业务状态
- 解释 LLM 输出
- 访问内部 Repository
- 通过前端逻辑补齐后端缺失能力
- 将招标书文件写入制度知识库

## 9. 前端阶段验收

### 工程验收

- 继续使用现有 `frontend` 工程。
- 不新建第二套前端包。
- TypeScript strict 检查通过。
- `npm run build` 通过。
- 统一请求、错误、上传和轮询行为。

### 功能验收

- 知识库可以完成上传、入库、检索、问答和引用查看。
- 文档失败状态可查看并重试。
- Tender Agent 可以完成上传、任务查询、骨架预览和下载。
- 页面刷新后状态可恢复。

### 视觉验收

- 采用 AetherFlow 的深色侧边栏、卡片和状态视觉语言。
- 工作区正文保持浅色、高可读性。
- 处理中、成功、失败状态颜色和文字明确。
- 桌面端优先，同时保证平板宽度可用。
