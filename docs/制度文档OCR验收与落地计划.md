# 制度文档 OCR 验收与落地计划

本文档用于把当前 OCR 能力从“已经接入”明确成“可验收、可继续推进”的状态，避免进入 `Milestone D` 后再回头补录入侧基础口径。

## 1. 当前结论

当前仓库里的 OCR 已不是空白能力，而是已经具备最小闭环：

- `.doc` 可先标准化为 `.docx` 再进入后续链路。
- 单张图片扫描件可直接进入 OCR。
- 图片式 DOCX 在缺少正文时可继续走 OCR。
- PDF 结构化解析阶段会识别疑似扫描件。
- 对无法直接提取正文的页面，会保留页级渲染入口。
- OCR 阶段会优先处理这类页级渲染图片块。
- OCR 后仍无有效正文时，入库模式会被 `ingest_guard` 拦截。

因此，当前 OCR 的核心问题已经不是“有没有”，而是：

- 哪些文档应该触发 OCR
- OCR 跑完后如何判断结果是否可接受
- 后续是继续扩批量，还是先沉淀样本验收集

## 2. 当前仓库内已具备的验收基础

本轮已补齐以下基础资产：

- `app/services/ingestion/steps/policy_ocr.py`
  - OCR 主服务实现
- `tests/test_policy_ocr.py`
  - 扫描 PDF 目标选择
  - OCR 禁用时的返回行为
  - OCR 成功后的正文回填和运行期元数据清理

这意味着 OCR 至少已经有一层“不会随手改坏主逻辑”的保护。

## 3. 当前建议的验收口径

建议把 OCR 验收拆成三档，而不是只看“有没有文本输出”。

### 3.1 A 档：可直接入库

满足以下条件：

- OCR 后形成连续正文
- 关键条款可读
- 标题、条款编号、核心实体没有大面积错乱
- 后续章节拆分与切块结果基本可用

### 3.2 B 档：需要人工复核

满足以下条件：

- OCR 后能形成正文，但有明显噪音
- 部分条款编号、标点或换行异常
- 仍可通过人工快速复核后继续入库

### 3.3 C 档：不建议直接入库

满足以下条件：

- OCR 后正文仍大面积缺失
- 条款顺序严重错乱
- 关键制度字段无法辨认
- 继续入库会直接污染知识库

## 4. 进入 Milestone D 前建议做到的最低要求

不需要现在就把 OCR 做成完整专项，但建议至少满足：

1. 保证扫描 PDF 在预览/入库链路里有统一行为。
2. 保证 OCR 禁用、缺少配置、OCR 失败三类场景都有明确返回。
3. 保证 OCR 成功后的文本来源和运行期图片元数据可追踪、可清理。
4. 留下一组后续可补充的真实样本验收清单。

## 5. 后续推进顺序

建议按下面顺序继续，而不是直接把 OCR 扩成大工程：

1. 先收集 5 到 10 份真实扫描制度样本。
2. 用当前链路跑一轮，按 A/B/C 三档记录结果。
3. 找出最影响入库质量的错误类型。
4. 再决定是调 OCR 提示词、补图像预处理，还是增加人工复核节点。

## 6. 当前一句话结论

当前 OCR 已经具备最小可用链路，本轮目标不是继续“发明 OCR”，而是先把它变成一项有测试保护、有验收口径、能支撑后续知识获取扩充的基础能力。
