# 制度样板首批入库方案

## 目标

第一阶段只处理 `制度规范与方法资料` 里的“非敏感、易解析、具有代表性”的制度样板，用来跑通首批知识库接入流程。

这一次的目标不是把所有制度资料一次性灌库，而是先沉淀一条稳定、可复用的制度入库路径：

1. 选出合格样板
2. 建立主档和版本记录
3. 提取全文
4. 清洗文本
5. 视情况拆章节
6. 最后再进入 RAG 切块和向量阶段

## 首批范围

首批只收下面这一类文件：

- 非敏感
- 原生电子版优先
- 文本结构相对完整
- 单份文件可以独立代表一项制度
- 适合映射到现有 `kb_policy_*` 表

推荐优先纳入的文件类型：

- `.doc`
- `.docx`
- `.pdf`
- `.txt`

推荐优先纳入的制度内容：

- 管理制度
- 审核制度
- 质量控制制度
- 业务档案管理制度
- 员工手册
- 保密制度
- 技术负责人制度
- 作业规范

## 首批排除规则

以下内容首批不进知识库：

- 图片文件：`.jpg`、`.jpeg`、`.png`
- 压缩包：`.zip`、`.rar`、`.7z`
- 纯扫描分页图
- 盖章版
- 含签字、签名、印章的文件
- 含身份证、联系方式、账号、密码等敏感信息的文件
- logo、照片、截图
- 空白模板和仅用于签署的附件

建议按文件名和路径先做硬过滤。命中以下关键词时，默认排除：

- `身份证`
- `签字`
- `签名`
- `公章`
- `法人章`
- `印章`
- `账号`
- `密码`
- `联系方式`
- `联系`
- `logo`
- `照片`
- `截图`
- `盖章`
- `空白`
- `模板`

## 为什么先做制度样板

`制度类资料` 是当前最适合先跑通的知识库对象，因为它同时满足三件事：

- 能直接映射到现有 [sql/001_kb_policy_schema.sql](D:/workspace/bid-document-agent/sql/001_kb_policy_schema.sql)
- 未来既能做全文检索，也能做版本管理
- 比人员证件、扫描图片、报价表更容易先做出稳定样板

## 和当前表结构的映射方式

### 1. 主档层：`kb_policy_document`

一项制度对应一条主档记录。

建议首批先填这些字段：

- `policy_code`
- `policy_name`
- `policy_category`
- `responsible_department`
- `status`

首批建议值：

- `policy_category` 固定为 `管理制度` 或更细的制度类别
- `status` 先写 `draft`

### 2. 版本层：`kb_policy_version`

每一份制度文件先作为一个版本记录入库。

建议首批先填这些字段：

- `policy_id`
- `version_seq`
- `version_label`
- `source_year`
- `source_path`
- `file_name`
- `file_ext`
- `file_hash`
- `is_scanned`
- `parse_method`
- `parser_status`
- `version_status`

首批建议值：

- `version_status = draft`
- `parser_status = pending`
- `parse_method = direct` 或 `ocr`

### 3. 正文层：`kb_policy_section`

不是所有制度在第一天都要拆章节。

建议策略：

- 原生 `doc/docx/txt`：优先抽全文，后拆章节
- 结构清晰的 `pdf`：抽全文后按标题规则拆章节
- 疑似扫描件：首批只保留版本记录，后续再 OCR

### 4. 检索层：`kb_policy_chunk`

不要在第一步就做全部切块和向量。

建议顺序：

1. 文件落版本表
2. 文本抽取成功
3. 基本文本清洗完成
4. 抽样人工校验
5. 再进行 chunk 和 embedding

## 首批实际操作顺序

### 第一步：先做候选清单

不要直接入库，先把制度候选文件筛出来。

目标是生成一份“制度样板候选清单”，至少包含：

- 原始路径
- 相对路径
- 文件名
- 扩展名
- hash
- 是否疑似扫描件
- 推荐解析方式
- 预估制度名称
- 排除原因或纳入原因

### 第二步：人工确认主档粒度

这里是第一批最重要的人工动作。

需要确认：

- 这份文件是不是一项独立制度
- 文件名是否需要清洗成正式制度名称
- 是否属于同一制度的不同版本

注意：

- 不要把“模板”“空白承诺书”“签署附件”当成制度主档
- 不要仅凭文件名自动断定版本关系

### 第三步：先入 `document + version`

确认完候选样板后，先只写入：

- `kb_policy_document`
- `kb_policy_version`

此时还不要求：

- 章节拆分完全正确
- 向量已经生成
- 审核流程全部跑完

### 第四步：补全文和清洗文本

解析完成后再回填：

- `raw_text`
- `clean_text`
- `page_count`
- `parser_status`

### 第五步：再做章节和 RAG

当首批样板的文本质量稳定后，再做：

- `kb_policy_section`
- `kb_policy_chunk`

## 推荐的首批样板规模

建议首批只做 `10-30` 份制度样板，不要一开始做满目录。

推荐优先顺序：

1. 原生 `docx`
2. 原生 `doc`
3. 可直接抽文本的 `pdf`
4. 疑似扫描 `pdf`

不推荐首批纳入：

- 分页图片
- 压缩包
- 带盖章版本
- 高度重复版本

## 建议的判断标准

如果一份制度文件同时满足下面四项，就很适合首批入库：

- 非敏感
- 不是图片
- 有完整标题
- 可以独立对应一条制度主档
