# 数据清洗学习笔记

在过往项目中如我处理过BI和 RAG, 在这涉及到了数据清洗在本次机会和 CODEX 对话加上本人的理解制作学习笔记，对“数据清洗”这个概念及其工程落地方式的理解，方便后续继续做知识库入库、RAG、文档解析时反复翻看。

## 一、什么是数据清洗

数据清洗，简单说就是：

**把原始数据处理成更规范、更可用、更可靠的数据。**

如果用工程化一点的话来说：

**数据清洗 = 发现并处理原始数据中的脏数据、噪音、格式不统一、缺失、重复、异常等问题，让它能被后续系统稳定使用。**

要注意一点：

- 数据清洗不是创造新数据
- 数据清洗也不是凭空修改业务含义
- 数据清洗的核心，是让原始数据变成系统真正“能吃”的样子

## 二、为什么需要数据清洗

只要数据不是完全由自己系统内部、严格受控地生成出来，通常就会有清洗需求。

最常见的原因有这些：

- 原始数据来源复杂
- 数据格式不统一
- 数据中混有噪音
- 数据质量不稳定
- 后续系统处理对数据规范性要求更高

比如在知识库入库场景里，一份 PDF 或 DOCX 抽出来的文本，往往并不是天然就适合直接入库的，可能会有：

- 页码
- 页眉页脚
- 多余空格
- 换行混乱
- 重复短句
- 标题和正文粘连
- 表格文本拼接不规整

如果不先做清洗，后面的章节拆分、结构化入库、检索、问答都会变得不稳定。

## 三、数据清洗的核心目标

数据清洗通常有三个核心目标：

1. 让数据更正确
2. 让数据更统一
3. 让数据更适合后续使用

可以进一步展开理解：

- 更正确：去掉明显错误、异常、重复、乱码
- 更统一：统一格式、统一类型、统一表达方式
- 更适合使用：为后续入库、检索、统计、建模、问答打基础

## 四、常见的数据清洗操作

日常项目里最常见的数据清洗操作，大体可以分成下面几类。

| 操作类型 | 典型问题 | 常见做法 |
|---|---|---|
| 去脏数据 | 空文件、页码、页眉页脚、测试数据、重复盖章件、无意义内容 | 规则过滤、正则匹配、黑名单关键词、长度阈值 |
| 补缺失值 | 字段为空、日期缺失、部门未填、来源信息不全 | 直接丢弃、给默认值、上下文推断、保留为 `NULL` |
| 格式统一 | 换行不统一、全角半角混用、多空格、扩展名大小写不一致、日期格式不统一 | `replace`、正则替换、`strip`、大小写统一、字符标准化 |
| 类型转换 | 字符串数字、字符串时间、文本状态值不能直接用于业务计算 | 数值转换、日期解析、布尔转换、枚举转换 |
| 去重 | 重复文档、重复 section、重复业务记录 | 按主键去重、按业务唯一键去重、按哈希去重、模糊去重 |
| 合法性校验 | 文件类型不允许、文本太短、页数异常、金额超范围、字段格式非法 | 类型白名单、大小阈值、长度阈值、正则校验、枚举约束 |
| 标准化枚举值 | 同一部门多种写法、同一状态多种表达、同一类制度命名不统一 | 映射表、标准命名规则、领域规则统一 |
| 结构化拆分 | 文档拆章节、表格拆行、全文拆段落、长文拆 section | 标题模式识别、按段落和换行拆分、按业务规则构建结构化对象 |

其中“结构化拆分”在文档知识库项目里尤其常见，因为原始文件通常是非结构化的。

## 五、Python 项目里常见的数据清洗实现方式

在 Python 项目里，数据清洗通常不是靠一个单独的大框架完成的，而是根据数据类型，组合使用不同手段。

常见实现方式大体有下面几类：

### 5.1 字符串处理

这是最基础、也最常见的一类。

常用手段：

- `replace`
- `strip`
- `split`
- `join`
- `lower`
- `upper`

适合处理：

- 多余空格
- 换行统一
- 前后空白去除
- 文件扩展名统一
- 字段名简单标准化

### 5.2 正则表达式

Python 里做文本型清洗时，`re` 几乎是最常用的工具之一。

常用手段：

- `re.sub`
- `re.match`
- `re.fullmatch`
- `re.findall`

适合处理：

- 页码识别
- 标题识别
- 手机号、邮箱、证件号格式判断
- 去除特定模式噪音
- 抽取结构化字段

### 5.3 列表 / 字典 / 集合循环处理

这类方式非常符合 Python 风格。

常用手段：

- 列表推导式
- 字典推导式
- `set` 去重
- `Counter` 计数
- `for` 循环过滤

适合处理：

- 重复值统计
- 批量规则过滤
- 短文本去重
- 行级清洗
- 分类映射

### 5.4 DataFrame 清洗

如果数据是表格型、结构化字段型，Python 里很常见的做法是使用 `pandas`。

常用手段：

- 缺失值处理
- 类型转换
- 批量替换
- 重复行处理
- 条件过滤

适合处理：

- Excel 导入
- CSV 清洗
- 统计分析前预处理
- 数据对账

### 5.5 文档解析后清洗

如果数据来源是 PDF、DOCX、网页、日志，Python 里通常是：

1. 先用解析库抽出原始内容
2. 再对抽出的文本做二次清洗

常见做法：

- PDF 先抽文本，再统一换行和空格
- DOCX 先抽段落和表格，再做噪音清理
- HTML 先去标签，再做正文抽取
- 日志先按行切分，再做正则筛选

### 5.6 规则函数组合

企业项目里比较常见的做法，不是写一个超大的“万能清洗器”，而是拆成多个小规则函数，再按顺序组合。

例如：

1. `normalize_whitespace`
2. `remove_page_number`
3. `remove_repeated_header`
4. `normalize_title`
5. `validate_min_length`

这样的好处是：

- 易读
- 易测
- 易扩展
- 易按业务阶段开关某条规则

## 六、Python 里常见的数据清洗相关库

Python 的优势之一，是不同数据形态都有比较成熟的处理生态。

| 类别 | 常见库 | 主要用途 |
|---|---|---|
| 纯文本和规则处理 | `re`、`collections`、`unicodedata`、`string` | 正则匹配、频次统计、Unicode 标准化、基础字符串处理 |
| 表格和结构化数据 | `pandas`、`openpyxl`、`csv` | Excel / CSV 清洗、字段批量转换、缺失值处理、去重与聚合 |
| 文档处理 | `python-docx`、`pypdf`、`pdfplumber`、`python-pptx` | DOCX 段落与表格抽取、PDF 文本抽取、页面级定位、PPT 文字抽取 |
| HTML 和网页数据 | `beautifulsoup4`、`lxml`、`readability-lxml` | 去标签、提取正文、清理网页噪音 |
| OCR 和图片文本 | `paddleocr`、`tesseract`、`easyocr` | 扫描件识别、图片转文本、票据/合同/盖章件预处理 |
| 校验和数据模型 | `pydantic`、`pydantic-settings` | 输入结构校验、默认值管理、字段约束、配置统一读取 |

## 七、数据清洗一般出现在哪些场景

数据清洗出现得非常广，几乎所有“数据不是自己完全严格生成出来”的场景里都会出现。

常见场景如下：

### 5.1 文件入库

比如：

- PDF
- DOCX
- Excel
- 邮件附件
- 扫描件

这类文件往往天然不规整，因此在正式入库前通常都要做清洗。

### 5.2 数据迁移

老系统迁新系统时，经常会碰到：

- 历史脏数据
- 枚举值不统一
- 重复记录
- 字段缺失

### 5.3 接口对接

第三方接口返回的数据可能不稳定、不规范、不完整，因此需要本系统接收后先做清洗和校验。

### 5.4 报表和 BI

做统计分析前，要先去掉重复值、空值、异常值，否则报表结果容易失真。

### 5.5 爬虫和网页采集

网页中的广告、菜单、标签、重复文本都属于噪音，必须先清洗。

### 5.6 日志处理

日志中可能会有无效行、敏感信息、格式不统一的内容，在进入分析系统前常常需要先做预处理。

### 5.7 机器学习 / NLP / RAG

模型、向量化、检索、问答都非常依赖输入数据质量，因此这类场景下数据清洗往往非常重要。

### 5.8 用户输入

表单、导入模板、后台录入的数据，经常带有：

- 多余空格
- 特殊字符
- 日期格式混乱
- 状态值乱填

这类场景本质上也属于数据清洗。

## 八、企业项目里常见的数据清洗应用方案

企业里做数据清洗，通常不会只写一个函数就结束，而是会形成一套相对固定的应用方案。

常见方案如下：

### 8.1 同步清洗

特点：

- 请求来了就立刻清洗
- 清洗成功后直接进入后续流程
- 适合低时延、小批量、单文件处理

常见场景：

- 单文件上传入库
- 表单提交
- 接口实时接收

优点：

- 逻辑直观
- 链路短
- 出错时反馈快

注意点：

- 不适合超大文件
- 不适合高并发重处理
- 需要控制单请求耗时和内存

### 8.2 异步清洗

特点：

- 请求先登记任务
- 后台任务再去解析、清洗、入库
- 前台通过任务状态轮询结果

常见场景：

- 大批量文件导入
- OCR
- 超大 PDF / DOCX 处理
- 多阶段知识库构建

优点：

- 不阻塞前台请求
- 更适合长流程和重处理
- 容易做失败重试

注意点：

- 需要任务状态管理
- 需要幂等控制
- 需要补偿和失败排查能力

### 8.3 分层清洗

特点：

- 输入层做轻校验
- 解析层做技术清洗
- 领域层做业务标准化
- 落库层只做约束和兜底

这是比较推荐的企业方案，因为更利于维护和扩展。

### 8.4 分阶段清洗

特点：

- 第一阶段只做保守清洗
- 第二阶段再做结构化增强
- 第三阶段再做检索或建模需要的深加工

常见场景：

- RAG
- 知识库建设
- 文档治理
- 数据治理平台

这种方式适合逐步落地 MVP，而不是一开始把所有复杂规则都堆进去。

### 8.5 可审计清洗

特点：

- 记录清洗前后结果
- 记录被删除的噪音样本
- 记录命中的规则
- 记录失败原因

适合场景：

- 企业制度入库
- 合规资料处理
- 财务数据治理
- 高可信知识库

这样做的价值在于：后续出了问题能追查。

## 九、企业端通常如何处理数据清洗

企业项目里，数据清洗往往不是一个“纯算法动作”，而是一个带工程约束的处理链路。

常见做法如下：

### 9.1 先定义输入边界

在真正做清洗前，先明确：

- 支持哪些文件类型
- 不支持哪些文件类型
- 文件大小上限是多少
- 哪些内容首期排除
- 哪些数据必须人工复核

这一步很重要，因为很多脏数据问题，最好的解决方式不是“硬清洗”，而是“禁止进入流程”。

### 9.2 先做轻量准入，再做重处理

企业里一般不会拿到文件就立刻全文解析，而是先做一层轻量准入判断：

- 文件是否存在
- 扩展名是否合法
- 文件是否过大
- 文件名是否命中排除规则
- 是否疑似扫描件

只有过了这一层，才进入真正的解析和清洗。

### 9.3 规则优先，模型辅助

在企业业务里，数据清洗第一选择通常仍然是规则，而不是一上来就交给模型。

原因：

- 规则稳定
- 成本低
- 可解释
- 可复现
- 易审计

模型通常适合用在：

- OCR
- 模糊分类
- 标题补全
- 异常识别
- 复杂字段抽取

但即便用了模型，前后通常也要配规则做约束。

### 9.4 小步迭代，不追求首版完美

企业端比较稳妥的做法通常是：

1. 先定义 MVP 范围
2. 先支持少量明确类型
3. 先做保守清洗
4. 先保证链路稳定
5. 再根据脏数据样本不断补规则

因为数据清洗高度依赖真实样本，很多规则不是坐在会议室里一次性设计出来的，而是在运行中不断补齐的。

### 9.5 失败要可见、可追踪、可重试

企业项目里，清洗失败不能只抛一个异常结束。

通常至少要能回答下面这些问题：

- 失败发生在哪个阶段
- 失败的是哪份文件
- 失败原因是什么
- 是否已经部分入库
- 能不能重试

如果是重要业务，还会进一步要求：

- 保留失败样本
- 保留原始文件路径
- 保留规则命中记录
- 保留人工复核入口

### 9.6 输出要服务后续环节

清洗结果不是“洗干净就完了”，还要考虑后面谁来消费。

例如：

- 入库层要什么字段
- 检索层要什么文本粒度
- embedding 层要不要保留原段落边界
- 问答层要不要保留章节路径
- 审核层要不要看到被删掉的噪音样本

也就是说，企业里的数据清洗一定是“面向下游”的。

## 十、Python 项目里常见的数据清洗实施细节

如果从 Python 工程实现角度看，一个相对通用的落地方式通常包括下面这些细节。

### 10.1 先定义输入 DTO

先把输入结构固定下来，例如：

- 文件路径
- 文件类型
- 来源信息
- 业务分类
- 可选人工标签

这样做的好处是：清洗逻辑不会直接依赖零散参数。

### 10.2 再定义输出结果对象

清洗结果不要只返回一个字符串，通常应该返回结构化结果，例如：

- 清洗后的正文
- 清洗备注
- 被移除的噪音示例
- 是否疑似异常
- 处理阶段状态

这样便于日志、预览、排错和后续落库。

### 10.3 每条规则尽量做成小函数

例如：

- `normalize_newlines`
- `normalize_whitespace`
- `remove_page_number`
- `remove_repeated_noise`
- `keep_heading_structure`

这样做的好处是：

- 单条规则容易测试
- 后期容易增删
- 容易判断是哪条规则导致误删

### 10.4 保守优先

在文档型项目里，首版清洗通常更适合：

- 宁可少删
- 不要误删正文
- 优先保留结构
- 先保证 section 能稳定拆分

这比一开始追求“极致干净”更适合企业落地。

### 10.5 规则顺序要稳定

清洗规则往往不是无序执行的。

例如更常见的顺序是：

1. 编码与换行标准化
2. 空白压缩
3. 页码和重复噪音移除
4. 标题和正文边界保护
5. 最终输出整理

如果顺序乱掉，可能会导致：

- 原本能识别的页码识别不到
- 原本要保留的标题被提前破坏
- 后续结构拆分不稳定

### 10.6 尽量保留中间结果

Python 项目里，实际落地时比较推荐保留：

- `raw_text`
- `clean_text`
- `removed_noise_examples`
- `notes`

这样对调试非常有帮助。

### 10.7 把规则和编排分开

推荐做法是：

- `service` 负责流程编排
- `cleaner` 负责清洗规则
- `domain rules` 负责业务判断
- `repository` 负责落库

这样不会让一个类既做解析、又做清洗、又做入库，导致越来越臃肿。

### 10.8 先写样本测试，再补规则

数据清洗最怕“感觉上没问题，实际一跑全是误删”。

因此比较推荐的做法是：

- 先准备几个真实样本文档
- 每次新增规则都补测试
- 保证典型文档不会被清洗坏

## 十一、企业项目里数据清洗的质量控制

企业端真正难的，不只是“把清洗写出来”，而是“怎么证明它可用、可控、可维护”。

通常会关注下面这些方面：

### 11.1 准确性

重点看：

- 是否删掉了不该删的正文
- 是否保留了关键结构
- 是否把明显噪音清掉了

### 11.2 稳定性

重点看：

- 同类文档是否表现一致
- 不同批次数据是否稳定
- 是否容易因个别格式变化失效

### 11.3 可解释性

重点看：

- 为什么这行被删了
- 为什么这个文件被拒绝
- 为什么这个标题被识别成 section

规则型清洗在这方面通常更有优势。

### 11.4 可回溯性

重点看：

- 原始文件路径是什么
- 原始文本是什么
- 清洗后文本是什么
- 命中了哪些规则
- 哪一步失败了

### 11.5 可扩展性

重点看：

- 新文件类型能不能接入
- 新规则能不能独立增加
- 是否能从单文件扩展到批量
- 是否能从保守清洗扩展到深度治理

## 十二、结合当前项目的一种典型实施方案

对于当前制度类知识库入库 MVP，可以用下面这种方案理解数据清洗落地：

1. 文件登记
2. intake 校验
3. 解析 DOCX / 原生 PDF
4. 产出 `raw_text`
5. 做保守清洗
6. 产出 `clean_text`
7. 按章 / 节 / 条拆 section
8. 一次性落库

其中保守清洗阶段通常做：

- 换行统一
- 全角空格转半角空格
- 连续空格压缩
- 典型页码移除
- 重复短噪音移除
- 保留章节结构

这类方案的特点是：

- 规则简单
- 成本可控
- 易解释
- 适合 MVP
- 便于后续逐步增强

## 十三、结合当前项目理解数据清洗

这一点非常重要，因为很多项目不是不会清洗，而是清洗逻辑放乱了。

一个更合理的思路是：

**数据清洗不应该全塞在一个地方，而应该按“技术清洗”和“业务清洗”分层放。**

### 6.1 API 层

适合放：

- 必填校验
- 基础类型校验
- 默认值填充
- 很轻量的字符串整理

不适合放：

- 大段文本清洗
- 文档解析
- 复杂业务规则

### 6.2 技术适配层 / 解析层

适合放“技术型清洗”：

- PDF 抽文本
- DOCX 抽段落
- 编码处理
- 换行统一
- 空格统一
- 页码初步处理

这类清洗更偏“让数据能被系统处理”。

### 6.3 领域层

适合放“业务型清洗”或“业务标准化”：

- 哪些文件属于制度类
- 哪些关键词应该排除
- 制度名称怎么规范化
- 章 / 节 / 条怎么识别
- 部门名称怎么统一

这类清洗更偏“让数据符合业务定义”。

### 6.4 应用服务层

适合放流程编排：

- 先解析
- 再清洗
- 再拆 section
- 再落库

这一层主要负责调度，不适合堆太多底层规则细节。

### 6.5 Repository / DB 层

适合放：

- 非空约束
- 默认值
- 唯一约束
- 类型约束

这一层更适合做兜底保护，不适合承载主清洗逻辑。

## 七、技术清洗和业务清洗的区别

这个区分对工程分层特别有帮助。

### 7.1 技术清洗

目标是：

**让数据能处理。**

常见内容：

- 格式统一
- 换行统一
- 去掉明显噪音
- 编码修正
- 类型转换
- 文件内容解析

### 7.2 业务清洗

目标是：

**让数据符合业务定义。**

常见内容：

- 制度类文件识别
- 标题识别规则
- 文件准入规则
- 版本标签生成规则
- 标准制度名称推导

### 7.3 一句话区分

- 技术清洗：让数据“能处理”
- 业务清洗：让数据“能入业务”

结合当前这个制度类知识库入库 MVP，可以把数据清洗理解为下面这条链路中的一环：

1. 文件登记
2. intake 校验
3. 格式标准化
4. 解析器选择
5. 原文抽取
6. 文本清洗
7. 章节拆分
8. 落库

其中和“清洗”最直接相关的内容包括：

- 原文抽取后的换行统一
- 空格统一
- 去除重复短噪音
- 去除典型页码
- 尽量保留章、节、条结构
- 为后续章节拆分提供更稳定的输入

要注意，这里的清洗策略是：

**保守清洗。**

也就是说：

- 不追求把文本洗得特别“漂亮”
- 更重视不误删正文
- 更重视后续结构化拆分稳定

## 十四、当前项目中的一个典型理解

以制度文档为例，原始文本可能像这样：

```text
第一章   总则

第 1 页

第一条   为规范公司采购行为...
```

经过保守清洗后，可能变成：

```text
第一章 总则

第一条 为规范公司采购行为...
```

这里做的就是典型的数据清洗：

- 空格统一
- 页码移除
- 保留制度结构

它的目的不是改写业务内容，而是让后面的 section 拆分更稳定。

## 十五、我当前对数据清洗的阶段性理解

结合这次沟通，我当前可以把“数据清洗”理解成下面这句话：

**数据清洗就是把外部来的、格式杂乱或质量不稳定的数据，按照技术规则和业务规则整理成可稳定处理、可稳定入库、可稳定复用的数据。**

对于当前知识库项目来说，它不是一个可有可无的小步骤，而是：

- 文档入库前的重要预处理
- 章节拆分前的重要稳定器
- 后续检索和问答质量的基础保障

## 十六、常见应用场景与对应处理方案

这一节不再只讲概念，而是按企业项目里最常见的场景来整理“通常怎么处理”。

| 场景 | 常见输入 | 常见脏数据 | 通常处理步骤 | Python 常用工具 | 企业项目注意点 |
|---|---|---|---|---|---|
| 文件入库 | PDF、DOCX、Excel、邮件附件、扫描件 | 页码、页眉页脚、多余空格、换行混乱、OCR 错字、重复扫描副本、文件名不规范 | 先做文件存在性/扩展名/大小校验，再判断是否允许入库；对可解析文件抽取原始文本；统一换行和空格；去除页码、重复短噪音、明显模板文字；保留标题、章节、表格等关键结构；输出清洗结果供 section 拆分或结构化入库 | `pathlib`、`python-docx`、`pypdf`、`pdfplumber`、`re` | 首版不要一下支持太多格式；扫描件和原生 PDF 要分开处理；重要文档建议保留 `raw_text` 和 `clean_text`；对删除内容保留示例便于复核 |
| Excel / CSV 导入 | 客户清单、供应商名单、项目台账、财务统计表 | 列名不统一、空单元格、日期格式混乱、数字列被读成字符串、重复行、合并单元格影响解析 | 先读取表头并做标准列名映射；检查必填列；处理空值、默认值、异常值；统一日期、金额、数量等字段类型；按主键或业务唯一键去重；生成结构化记录后再落库或对账 | `pandas`、`openpyxl`、`csv`、`datetime` | 模板导入最好固定表头；不要直接相信用户上传的类型；金额、税率、日期列最容易出问题；批量导入最好输出错误明细 |
| 第三方接口对接 | 外部平台回调、第三方业务接口、供应商系统同步、政务平台返回结果 | 字段缺失、枚举值不一致、时间格式不统一、布尔表达不一致、空串代替 `null`、数字字段返回文本 | 先校验响应结构；将第三方字段映射为内部字段；统一时间、状态、金额、布尔类型；对异常字段做默认值或拒绝处理；保留原始响应和清洗后结果；再进入内部业务逻辑 | `pydantic`、`requests`、`httpx`、`datetime`、`decimal` | 外部接口经常变更字段，需要兼容旧值；映射规则最好单独维护；高风险接口建议保留原始报文；对状态值和枚举值要建立标准字典 |
| 数据迁移 | 老系统 SQL、Excel 批量数据、CSV 历史数据、多系统合并数据集 | 历史空值、重复记录、编码不一致、枚举值不统一、主键冲突、历史脏备注字段 | 先梳理原始字段和目标字段映射；标准化编码、日期、状态、枚举值；按业务主键识别重复数据；对无效数据做隔离或单独导出；迁移前抽样验证；迁移后做数量、金额、关键字段对账 | `pandas`、`sqlalchemy`、`csv`、`json` | 数据迁移最怕值不对但看起来导进去了；必须做迁移前后对账；不仅要关注能否导入，还要关注是否符合新系统业务定义；历史脏数据要单独留痕 |
| RAG / 知识库 | 制度文件、操作手册、会议纪要、合同模板、FAQ 文档 | 标题层级不稳定、OCR 噪音、长段落缺少边界、表格抽取后变形、同一制度多个版本混在一起、大段重复模板文字 | 先筛掉不适合首期入库的文件；抽取原始文本并判断是否疑似扫描件；统一换行、空格和页面噪音；尽量保留章节结构和标题路径；拆成 section 或 chunk；再进入 embedding、检索、问答等后续环节 | `python-docx`、`pypdf`、`pdfplumber`、`re`、`pydantic` | 不要过早为了 chunk 破坏原始结构；制度类文档要优先保留章、节、条；首版适合做保守清洗；清洗策略要为后续检索和引用服务 |
| 用户表单输入 | 后台新增页面、Excel 导入模板、外部用户注册资料、审批流表单 | 前后空格、手机号带分隔符、日期手填格式不统一、必填项漏填、状态值乱填、多选字段分隔符混乱 | 先做必填和格式校验；统一空白和分隔符；统一手机号、邮箱、日期等字段格式；对枚举字段映射为标准值；无法纠正的输入直接返回错误提示 | `pydantic`、`re`、`datetime`、`email-validator` | 表单清洗不能悄悄篡改用户真实意图；能自动纠正的轻问题可以纠正；影响业务含义的问题必须显式报错；错误提示要具体 |
| 日志与监控数据 | 应用日志、审计日志、接口调用记录、异常堆栈 | 无效空行、日志格式不统一、敏感信息未脱敏、同一异常大量重复、多行堆栈难以归并 | 先按日志格式切分记录；抽取时间、级别、模块、错误码等字段；过滤空行和无效记录；做敏感信息脱敏；对重复异常聚合或计数；输出适合检索或告警的数据结构 | `logging`、`re`、`json`、`pandas` | 日志清洗不能破坏排障信息；脱敏规则要优先于聚合和展示；监控场景要控制日志量；对关键错误要保留足够上下文 |

## 16.8 一个通用判断方法

如果后面遇到新的业务场景，可以先按下面这个顺序判断怎么设计清洗方案：

1. 输入数据是什么类型
2. 主要脏数据是什么
3. 后续谁要消费清洗结果
4. 哪些错误可以自动修正
5. 哪些错误必须直接拒绝
6. 是否需要保留原始数据和中间结果
7. 是否需要同步处理还是异步处理

只要先把这几个问题想清楚，数据清洗方案通常就不会偏得太远。
